## 先决定什么不发布

个人档案站最容易出现的问题不是内容太少，而是把本机路径、测试账号、密钥、私人文档或未经复核的截图一起提交到公开仓库。

公开边界应当在写作前确定，而不是发布后再搜索敏感文本。

## 内容分级

| 级别 | 例子 | 处理 |
| --- | --- | --- |
| A / 可公开 | 项目目标、通用技术栈、已验证阶段 | 可以直接进入 Content Collections |
| B / 降级公开 | 私有项目的方法、未公开界面的概述 | 只保留问题域与技术方法 |
| C / 暂缓 | 缺少事实支撑的项目名、未脱敏截图 | 等待补充或复核 |
| D / 不公开 | 密钥、身份信息、私有数据与地址 | 不进入仓库 |

## 截图检查

截图发布前至少检查：

1. 地址栏、窗口标题和标签页。
2. 用户目录、文件路径和终端提示符。
3. 通知、聊天、邮箱和账号头像。
4. 文档正文、论文批注和测试数据。
5. 内网域名、端口和调试面板。

## 仓库规则

私有草稿只有两种可接受状态：不进入仓库，或放在明确被 Git 忽略的位置。`draft: true` 只控制站点构建，不等于内容没有被提交。

```txt
draft: true  -> not rendered
gitignored   -> not committed
```

两者解决的是不同风险。

## 发布前问题

> 如果这段内容脱离当前上下文被单独看到，它是否仍然适合公开？

这个问题比“页面上是否显示”更严格，也更接近长期维护公开档案的真实需求。
