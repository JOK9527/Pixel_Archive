> 执行日期：2026-06-12

## 1. 目标

在继续扩充站点内容前，先确认哪些信息可以公开、哪些信息需要降级描述、哪些内容必须等待事实补充。后续 Phase 12–14 只使用本清单中已确认的材料，不以占位文案冒充真实成果。

| A | Pixel Archive | 建站目标、Astro / TypeScript / MDX / CSS 技术栈、内容集合、视觉系统与当前阶段 | 可直接发布并作为 Featured Project |

### Projects 脱敏规则

- 不写真实密钥、Token、Cookie、账号、内网地址、私有仓库地址。
- 不公开本机绝对路径、用户目录名、测试文件路径或带身份信息的截图。
- 未经过复核的性能、准确率、用户量和完成度不写为确定事实。
- 私有项目只描述可公开的问题域与技术方法，不披露客户、机构或数据来源。

## 3. Notes 清点

Phase 13 首批 Notes 使用可从仓库和提交历史核验的主题：

1. Content Collections 初步整理：现有内容集合与引用关系。
2. Quiet Pixel Archive 视觉系统再校准：颜色层级、像素组件语言与约束。
3. 分层背景的硬边界排查：CSS 渐变尺寸、重复方式与暗色模式差异。
4. 个人档案站的公开边界：内容分级、脱敏与发布前检查。
5. 项目详情页的信息组织：状态、技术栈、时间线、关联笔记与返回路径。

Notes 可以包含代码片段、表格和公式样式示例，但示例必须服务于实际主题，不加入无来源的结论。

## 4. Archive 清点

Phase 14 首批 Archive 使用已发生且可从仓库验证的保存点：

1. Pixel Archive 项目启动。
2. 内容集合与基础路由完成。
3. Phase 10 视觉系统再校准完成。
4. Phase 19 色彩层级调整完成。
5. Phase 11 真实内容边界确认。

Archive 只记录时间、发生事项、影响范围和关联内容，不将计划中的工作写成已完成。

## 5. Lab 清点

Phase 14 首批 Lab 保留为可复查的界面实验：

1. Pixel UI Study Slot：像素角标、分割线、状态块的组合规则。
2. Project Cover System：文件块、终端块、存档芯片和像素网格的抽象封面组合。

Lab 条目必须标明 `idea`、`wip` 或 `done`，避免实验草图与正式功能混淆。尚未制作的插图、Minecraft 场景或复杂动效只列为候选，不展示伪造结果。

## 6. About 公开边界

首版 About 只公开：

- Pixel Archive 的用途与维护原则。
- 关注的公开技术方向：Web 工具、阅读与知识整理、界面系统。
- 站内 Projects、Notes、Archive、Lab 的内容关系。
- “持续整理中”的真实状态。

首版不公开：

- 真实姓名、学校、单位、住址、手机号、私人邮箱。
- 未确认的社交账号或外部链接。
- 可推断个人日程、地理位置或设备信息的细节。
- 未取得公开许可的合作方、客户、文档和数据。

## 7. 图片与附件边界

- 当前优先使用站内 CSS / SVG 抽象视觉，不以临时截图填充项目封面。
- 后续真实截图发布前检查窗口标题、地址栏、文件路径、用户名、通知和文档内容。
- 无法完成脱敏的截图不进入 `public` 或 Content Collections。
- 图片替代文本描述画面信息，不写无意义的“图片”或“截图”。
