建议位置：`docs/phases/20_Phase11-15_后续开发计划.md`

## 0. 项目源信息复读

在制定后续计划前，必须先复读 Pixel Archive 的项目源信息，避免后续开发偏离原始定位。

### 0.1 项目定位

Pixel Archive 是一个个人网站项目，但不是传统简历页，也不是普通博客模板。

项目核心定位是：

```txt
Pixel Archive 是一个以极简阅读体验为底层、以像素组件为识别符号、以复古终端 / 存档隐喻为氛围的个人档案馆。
```

长期用于存放和展示：

```txt
项目作品
技术笔记
开发日志
阶段性存档
视觉 / 像素 / Minecraft / 图像实验内容
```

### 0.2 核心设计理念

```txt
极简做骨架
像素做识别
复古做氛围
内容可长期沉淀
结构可持续扩展
```

理解方式：

```txt
极简负责：阅读、留白、信息层级、长期维护
像素负责：按钮、边框、标签、首页识别、404、Lab
复古负责：archive、save slot、terminal、save point 等隐喻
```

网站可以有个人风格，但不能牺牲阅读体验和信息清晰度。

### 0.3 初版页面范围

导航固定为：

```txt
Home / Projects / Notes / Archive / Lab / About
```

同时实现：

```txt
404 页面
```

各页面职责：

```txt
Home：私人像素档案终端入口，不是个人简介页
Projects：项目与作品展示
Notes：文章、学习笔记、开发日志
Archive：重要阶段和项目存档点
Lab：视觉、像素、Minecraft、图像、小工具等实验内容
About：极简个人入口与联系方式，不做正式简历页
404：lost archive / save file not found 概念
```

### 0.4 技术路线复读

初版技术栈：

```txt
Astro
TypeScript
MDX
Astro Content Collections
原生 CSS
CSS Variables
```

技术原则：

```txt
静态优先
内容优先
依赖克制
无后端
无 CMS
无数据库
本地先跑通
后续部署到 Cloudflare Pages
```

明确不使用：

```txt
Tailwind CSS
shadcn/ui
完整第三方 UI 框架
React / Vue UI 组件库
后端服务
数据库
CMS
复杂动画库
```

### 0.5 内容管理路线复读

内容通过 Markdown / MDX 文件维护。

推荐目录：

```txt
src/content/
├─ projects/
├─ notes/
├─ archive/
└─ lab/
```

图片统一放在：

```txt
public/images/
├─ projects/
├─ notes/
├─ archive/
├─ lab/
└─ shared/
```

维护方式：

```txt
本地 Markdown / MDX
Git 管理版本
后续可从 Obsidian 迁移
不做在线 CMS
```

## 1. 后续阶段总原则

Phase 10 之后不应继续盲目扩张视觉系统，而应转向：

```txt
真实内容基线
内容模板稳定
详情页阅读体验
站内发现能力
发布与长期维护
```

阶段顺序建议：

```txt
Phase 11：真实内容清点与公开边界
Phase 12：Projects 真实内容填充与项目详情增强
Phase 13：Notes 内容填充与 Markdown 阅读体验增强
Phase 14：Archive / Lab / About 内容成型
Phase 15：站内发现、SEO、RSS、部署发布增强
```

不建议现在做：

```txt
评论系统
登录系统
数据库
CMS
后端接口
GitHub API 自动同步
复杂像素房间首页
大规模动画系统
多语言系统
```

## 2. Phase 11：真实内容清点与公开边界

### 2.1 阶段目标

把后续要公开展示的真实内容先整理清楚，避免一边写页面一边暴露不该公开的信息。

本阶段不是大量写内容，也不是大改页面，而是建立真实内容的输入清单和公开边界。

### 2.2 必读文档

```txt
docs/00_项目开发计划书.md
docs/01_规范总览指南.md
docs/standards/04_内容与文件管理规范.md
docs/standards/07_文案与内容风格规范.md
docs/standards/15_数据模型与内部接口规范.md
docs/standards/17_公开信息与隐私边界规范.md
docs/phases/20_Phase11-15_后续开发计划.md
```

### 2.3 任务范围

建立内容候选清单：

```txt
候选 Projects
候选 Notes
候选 Archive 节点
候选 Lab 条目
About 可公开信息
需要隐藏或模糊处理的信息
需要后续补图的信息
```

建议输出文件：

```txt
docs/phases/21_Phase11_真实内容清点与公开边界.md
```

可选新增内部文件：

```txt
docs/private-drafts/内容候选清单.md
```

如使用 `private-drafts/`，必须确认该目录是否需要加入 `.gitignore`，避免误提交隐私内容。

### 2.4 输出标准

Phase 11 完成后应得到：

```txt
1. 公开项目清单
2. 公开笔记清单
3. 存档节点清单
4. Lab 实验候选清单
5. About 页面公开信息边界
6. 需要脱敏或暂不公开的信息列表
7. 后续内容填充优先级
```

### 2.5 禁止事项

不得：

```txt
直接把真实姓名、手机号、私人邮箱、学校/导师/项目敏感信息放到前台
直接提交未确认公开边界的内容
为了填充页面而编造经历
把所有聊天记录或开发日志原样公开
引入 CMS 或数据库
```

### 2.6 验收标准

```txt
npm run build 通过
前台页面不一定大改
内容边界清单明确
后续可进入真实内容填充
没有新增隐私泄漏风险
工作区改动可解释
```

建议提交信息：

```bash
git commit -m "docs: plan real content boundaries"
```

## 3. Phase 12：Projects 真实内容填充与项目详情增强

### 3.1 阶段目标

优先让 Projects 页面从占位状态进入可展示状态。

Projects 是个人网站最容易形成展示价值的页面，应优先填入 2–4 个真实项目，而不是一次性追求全量内容。

### 3.2 内容优先级建议

优先项目：

```txt
1. Pixel Archive 本站本身
2. ScholarLens AI
3. GLM-OCR / 文献解析相关工具
4. 舵机通用测试平台 / 系统辨识相关项目
5. 其他飞控 / AI / 工具类项目候选
```

每个项目至少包含：

```txt
标题
description
status：WIP / Done / Prototype / Paused
category
tags
featured 是否精选
startDate / updatedDate
summary
tech stack
项目亮点
当前状态
下一步
公开链接占位或 GitHub 链接占位
```

### 3.3 页面能力增强

允许增强：

```txt
Project Detail 信息层级
项目状态 Badge
项目技术栈展示
项目时间线小节
项目截图占位位
项目相关 Notes 链接占位
返回列表入口
```

不建议本阶段做：

```txt
GitHub API 自动拉取 stars / commits
复杂项目筛选系统
项目对比表
大量动画
复杂项目封面生成系统
```

### 3.4 文件范围

可能涉及：

```txt
src/content/projects/*.mdx
src/pages/projects/[slug].astro
src/pages/projects/index.astro
src/components/projects/*
public/images/projects/*
```

### 3.5 验收标准

```txt
至少 2 个真实 Projects 可展示
至少 1 个 featured project
项目详情页可阅读
所有项目内部链接可用
没有暴露隐私内容
移动端 Projects 列表和详情正常
npm run build 通过
控制台 0 error / 0 warning
```

建议提交信息：

```bash
git commit -m "content: add initial real project entries"
```

## 4. Phase 13：Notes 内容填充与 Markdown 阅读体验增强

### 4.1 阶段目标

让 Notes 页面具备真正的阅读价值，并补齐长文阅读基础能力。

Notes 是长期沉淀的核心，必须保证阅读体验，而不是追求强像素化。

### 4.2 内容优先级建议

先放 3–5 篇轻量真实内容：

```txt
1. Pixel Archive 开发日志
2. ScholarLens AI 阶段复盘
3. Astro Content Collections 使用记录
4. GLM-OCR / PDF 解析方案记录
5. 飞控 / GNN / 工具链学习笔记候选
```

每篇 Notes 至少包含：

```txt
title
description
type：article / log / note
category
tags
date
updated
summary
正文
```

### 4.3 Markdown / MDX 增强范围

允许增强：

```txt
标题层级 h2 / h3 / h4
段落行距
列表
引用块
代码块
内联代码
图片说明
表格
公式基础样式
TOC 基础体验
上一篇 / 下一篇链接
返回 Notes 列表
```

谨慎增强：

```txt
代码高亮主题
公式渲染依赖
自动目录插件
脚注
Callout 块
```

若引入任何新依赖，必须先说明理由，并保持依赖克制。

### 4.4 不做事项

本阶段不做：

```txt
全文搜索
评论系统
点赞收藏
Obsidian 自动同步
复杂双栏阅读器
多语言切换
```

### 4.5 文件范围

可能涉及：

```txt
src/content/notes/*.mdx
src/pages/notes/[slug].astro
src/pages/notes/index.astro
src/components/notes/*
src/components/markdown/*
src/styles/markdown.css
public/images/notes/*
```

### 4.6 验收标准

```txt
至少 3 篇真实 Notes 可阅读
Notes 列表显示类型 / 标签 / 日期
详情页排版清晰
代码块和表格不破版
移动端阅读正常
暗色模式阅读正常
npm run build 通过
控制台 0 error / 0 warning
```

建议提交信息：

```bash
git commit -m "content: add initial notes and improve reading styles"
```

## 5. Phase 14：Archive / Lab / About 内容成型

### 5.1 阶段目标

补齐非 Projects / Notes 的三类页面，让网站从“项目和笔记站”变成真正的个人档案馆。

### 5.2 Archive

Archive 应记录重要阶段，不记录太碎的日常内容。

建议首批节点：

```txt
Started Pixel Archive
Phase 9 第一阶段验收
Phase 10 视觉系统再校准
ScholarLens AI 某阶段完成
毕业设计 / 系统辨识阶段收尾
重要工具原型完成
```

允许增强：

```txt
按年份 / 月份分组
节点状态
相关 Project / Note 链接
save point 文案
```

暂不做：

```txt
Archive 详情页
过细日记流
复杂时间轴动画
```

### 5.3 Lab

Lab 是实验区，可以比其他页面更自由，但仍应控制复杂度。

首批 Lab 候选：

```txt
Pixel UI Study Slot
配色系统实验记录
Minecraft 建筑 / 像素实验占位
图像生成视觉实验占位
小型 UI 组件实验
```

允许增强：

```txt
Lab 卡片
实验状态
小型预览图
相关 Note / Project 链接
```

暂不做：

```txt
大型交互 Demo
Canvas 游戏
复杂像素房间
外部 API 实验
```

### 5.4 About

About 继续保持克制。

建议内容：

```txt
一句简短定位
项目 / 笔记 / 实验的说明
GitHub 链接
邮箱或联系方式占位
隐私边界说明
```

不得做成：

```txt
正式简历页
求职落地页
过度私人化介绍
长篇自传
```

### 5.5 文件范围

可能涉及：

```txt
src/content/archive/*.mdx
src/content/lab/*.mdx
src/pages/archive/index.astro
src/pages/lab/index.astro
src/pages/about.astro
src/components/archive/*
src/components/lab/*
public/images/archive/*
public/images/lab/*
```

### 5.6 验收标准

```txt
Archive 至少 4 个真实节点
Lab 至少 2 个实验条目
About 有可公开的极简信息
页面之间有合理内部链接
无隐私泄漏
移动端正常
暗色模式正常
npm run build 通过
控制台 0 error / 0 warning
```

建议提交信息：

```bash
git commit -m "content: expand archive lab and about pages"
```

## 6. Phase 15：站内发现、发布与长期维护增强

### 6.1 阶段目标

当真实内容达到一定数量后，再做站内发现与发布增强。

本阶段不是内容填充本身，而是让已有内容更容易被访问、索引、维护和发布。

### 6.2 启动条件

建议满足以下条件后再进入：

```txt
Projects >= 3
Notes >= 5
Archive 节点 >= 5
Lab 条目 >= 2
内容标题、描述、标签基本稳定
```

如果内容量不足，Phase 15 应延后。

### 6.3 可做功能

按优先级：

```txt
1. 标签 / 分类聚合页
2. Related content 基础关联
3. RSS
4. 基础 sitemap
5. Open Graph / meta 增强
6. Pagefind 搜索
7. Cloudflare Pages 正式部署
8. 正式域名绑定
```

### 6.4 Pagefind 搜索边界

搜索功能在计划书中属于后续路线图，不属于第一阶段强制范围。

可采用 Pagefind 的前提：

```txt
项目已经是静态站
内容量已经值得搜索
不需要后端
不引入重型搜索服务
```

不得做：

```txt
自建搜索后端
数据库搜索
Algolia 等外部服务优先接入
复杂搜索语法
个性化推荐
```

### 6.5 SEO / OG 边界

允许：

```txt
title / description
canonical
sitemap
RSS
基础 Open Graph 图片
每页 meta
robots.txt
```

暂不做：

```txt
复杂自动 OG 图片生成服务
多语言 SEO
站点统计追踪系统
广告 / 商业转化
```

### 6.6 部署边界

部署路线仍遵循计划书：

```txt
GitHub 仓库
Cloudflare Pages
后续考虑正式域名
```

不急于购买域名。正式域名前应先确认：

```txt
公开内容无隐私问题
README 清晰
构建稳定
404 正常
所有内部链接正常
```

### 6.7 验收标准

```txt
npm run build 通过
静态页面全部生成
站内链接正常
RSS / sitemap 如启用则可访问
搜索如启用则可索引当前内容
meta 信息合理
Cloudflare Pages 预览部署成功
移动端和暗色模式无明显问题
控制台 0 error / 0 warning
```

建议提交信息：

```bash
git commit -m "feat: improve content discovery and publishing readiness"
```

