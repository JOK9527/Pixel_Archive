# SKILL.md｜项目规范体系生成与执行流程

版本：v1.0  
适用场景：从一个新项目想法出发，逐步确定项目定位、架构、技术路线、视觉方向、内容体系、开发规范、Agent 执行规范，并生成一套可长期维护的 Markdown 规范文档体系。  
核心目标：避免一开始就直接写代码或堆大文档，而是先把“项目是什么、怎么做、哪些不能做、Agent 怎么执行、怎么验收”固定下来，再分阶段开发。  

---

## 1. 这个 Skill 解决什么问题

当用户准备开始一个新项目时，常见问题是：

```txt
1. 项目还没想清楚就开始写代码
2. 技术栈、页面结构、视觉方向、内容体系混在一起讨论
3. 一次生成一个超大总文档，后续 Agent 读不动
4. 规范重复、冲突、没有主定义归属
5. 开发 Agent 容易擅自扩展、超阶段开发
6. README、WORKLOG、验收、Git 节奏没有提前约束
7. 项目公开上线时忽略隐私、路径、API Key、截图风险
```

本 Skill 的目的：

```txt
先确定项目架构
再确定规范体系
再逐份生成文档
再更新总览
最后才进入 Agent 开发
```

---

## 2. 使用原则

执行此 Skill 时必须遵守：

```txt
1. 先问清项目定位，再写规范
2. 先确定架构，再填充细节
3. 先搭文档目录，再逐个生成文件
4. 每份文档只负责一个主题
5. 不把所有规则塞进一个大文件
6. 总览文档只做导航，不重复所有细节
7. 参考资料不等于强规范
8. 后续功能池不等于当前任务清单
9. Agent 指令模板和 Agent 工作流要分开
10. 每次生成文件后提供下载链接
```

---

## 3. 推荐总流程

完整流程分为 8 步：

```txt
Step 0：收集项目基本信息
Step 1：确定项目定位和边界
Step 2：确定技术路线和总体架构
Step 3：确定页面 / 模块 / 内容体系
Step 4：设计规范文档体系
Step 5：逐份生成规范文档
Step 6：补充关键强规范
Step 7：更新总览文档
Step 8：进入 Agent 开发阶段
```

---

## 4. Step 0：收集项目基本信息

在正式写规范前，先收集：

```txt
项目名称
项目类型
目标用户
项目用途
是否公开
是否需要部署
是否需要后端
是否需要数据库
是否需要登录
是否需要长期维护
是否需要 Agent 参与开发
用户偏好的视觉风格
用户偏好的技术栈
用户不想使用的技术
已有参考网站 / 模板 / 项目
已有内容来源
```

如果用户已经在对话中给出足够信息，不要重复问。  
如果关键信息缺失，优先根据上下文给出稳妥默认方案。

---

## 5. Step 1：确定项目定位和边界

先写清楚：

```txt
这个项目是什么
这个项目不是什么
第一阶段做什么
第一阶段不做什么
后续可以做什么
哪些功能明确延后
```

示例输出：

```txt
项目是一个个人档案网站
不是传统简历页
不是 CMS
不是后端应用
第一阶段只做静态站骨架、页面、内容系统、视觉基础
搜索、评论、RSS、SEO、API 等后续再做
```

这一阶段的重点是“收窄范围”，不是扩展功能。

---

## 6. Step 2：确定技术路线和总体架构

在生成大量规范之前，必须先确定：

```txt
技术栈
部署方式
目录结构
内容管理方式
页面结构
模块边界
数据流
内部接口意识
开发阶段
```

对于静态个人网站，推荐先明确：

```txt
Astro / Next.js / Vue / 其他
TypeScript 是否使用
Markdown / MDX 是否使用
Content Collections / 文件内容系统
CSS 方案
是否使用 UI 框架
是否需要后端
是否需要数据库
部署到哪里
```

注意：

```txt
像“确定项目架构”这类内容，应该一开始就做。
后续所有规范都是对这个架构的填充、细化和约束。
```

---

## 7. Step 3：确定页面 / 模块 / 内容体系

在文档生成前，先确定：

```txt
有哪些页面
有哪些模块
有哪些内容类型
每类内容有哪些字段
每个页面展示什么
每个模块负责什么
哪些内容可以为空
哪些功能只是预留
```

示例：

```txt
Home
Projects
Notes
Archive
Lab
About
404
```

内容类型示例：

```txt
projects
notes
archive
lab
```

模块示例：

```txt
layout
home
project
notes
archive
lab
ui
content
styles
data
```

## 8. Step 4：设计规范文档体系

不要一开始就生成所有内容。  
先输出文档体系目录，让用户确认。

推荐结构：

```txt
docs/
├─ 00_项目开发计划书.md
├─ 01_规范总览指南.md
│
├─ standards/
│  ├─ 02_技术与开发执行规范.md
│  ├─ 03_视觉风格规范.md
│  ├─ 04_内容与文件管理规范.md
│  ├─ 05_模块化架构规范.md
│  ├─ 06_组件设计规范.md
│  ├─ 07_文案与内容风格规范.md
│  ├─ 08_部署与运维规范.md
│  ├─ 09_验收与测试规范.md
│  ├─ 15_数据模型与内部接口规范.md
│  ├─ 16_阶段任务拆分与里程碑规范.md
│  └─ 17_公开信息与隐私边界规范.md
│
├─ agent/
│  ├─ 10_Agent执行指令模板.md
│  └─ 11_Agent工作流规范.md
│
└─ references/
   ├─ 12_视觉参考资源清单.md
   ├─ 13_技术参考资源清单.md
   └─ 14_后续功能规划池.md
```

说明：

```txt
00–01：项目入口与规范导航
02–09：核心专项规范
10–11：Agent 执行层
12–14：参考资料与规划池
15–17：关键补充强规范
```

## 9. Step 5：逐份生成规范文档

用户确认文档体系后，不要一次性生成所有文件。  
按编号逐份生成：

```txt
00
01
02
03
...
17
```

每次生成时：

```txt
1. 简短说明本文件负责什么
2. 不重复其他文件的大段内容
3. 写入 /mnt/data/xxx.md
4. 输出下载链接
5. 等用户继续
```

生成时要遵守：

```txt
总览文档只做导航
专项规范只管专项主题
Agent 模板只放可复制指令
参考资源只做参考，不做强约束
后续功能池不等于当前开发任务
```

## 11. Step 6：补充关键强规范

在生成 `00–14` 后，检查是否还缺少以下关键约束：

```txt
数据模型是否足够清晰
阶段边界是否足够清晰
公开信息和隐私边界是否足够清晰
```

如果缺少，应补：

```txt
15_数据模型与内部接口规范.md
16_阶段任务拆分与里程碑规范.md
17_公开信息与隐私边界规范.md
```

这三份应放入：

```txt
docs/standards/
```

不要放入：

```txt
docs/references/
docs/agent/
```

因为它们是强约束。

---

## 12. Step 7：更新总览文档

新增规范后，必须更新 `01_规范总览指南.md`。

至少更新：

```txt
文档体系总结构
每份文档职责
主定义归属表
不同任务阅读清单
Agent 开发前检查
当前阶段边界
规范维护规则
```

更新后要确认总览文档里包含新增文件名，例如：

```txt
15_数据模型与内部接口规范.md
16_阶段任务拆分与里程碑规范.md
17_公开信息与隐私边界规范.md
```

如果下载链接出现旧版缓存或同名文件混乱，应额外生成：

```txt
01_规范总览指南_vX.X_含新增规范.md
```

并明确让用户下载新版。

## 13. Step 8：进入 Agent 开发阶段

规范体系完成后，再让 Agent 开发。

推荐做法：

```txt
1. 先把 docs/ 文件放入项目
2. 让 Agent 先读 00 和 01
3. 按当前阶段读取对应 standards
4. 使用 10_Agent执行指令模板.md 里的任务模板
5. 要求 Agent 按 11_Agent工作流规范.md 汇报
6. 每轮执行 npm run build
7. 更新 README 和 WORKLOG
8. 用户验收后再 commit
```

不要一开始就对 Agent 说：

```txt
帮我把网站做好。
```

应该说：

```txt
请完成 Phase 0：项目初始化。开发前阅读 00、01、02、09、16。只做本阶段内容，完成后执行 npm run build 并更新 README / WORKLOG。
```

## 14. 文件生成规则

生成 Markdown 文件时：

```txt
1. 文件名使用中文编号格式
2. 写入 /mnt/data/
3. 生成后输出 sandbox 下载链接
4. 如果同名文件可能缓存，生成带版本号文件
5. 必要时读取文件内容确认关键字段存在
```

示例：

```txt
/mnt/data/01_规范总览指南.md
/mnt/data/01_规范总览指南_v1.2_含15-17.md
```

## 15. 生成每份文档时的回复格式

生成前简短说明：

```txt
继续生成 `05_模块化架构规范.md`。这份只管模块划分、模块职责、依赖方向、数据流和禁止耦合，不重复视觉、内容命名或组件 props 细节。
```

生成后回复：

```txt
已生成：

[下载 05_模块化架构规范.md](sandbox:/mnt/data/05_模块化架构规范.md)
```

不要在 final 里复制整份文件正文。

## 16. 质量检查清单

每份文档生成后检查：

```txt
1. 是否只负责自己的主题
2. 是否与其他文档职责分离
3. 是否有清晰用途
4. 是否有适用场景
5. 是否有禁止事项
6. 是否有验收标准或使用规则
7. 是否没有过多重复其他文档
8. 是否文件名正确
9. 是否提供下载链接
10. 是否需要更新 01_规范总览指南.md
```

总览文档生成后额外检查：

```txt
1. 是否列出全部文档
2. 是否包含新增文档
3. 是否结构路径正确
4. 是否职责清楚
5. 是否主定义归属明确
6. 是否阅读清单覆盖常见任务
```

## 17. 遇到用户质疑时的处理

如果用户指出：

```txt
文件内容不对
缺少新增文档
下载链接还是旧版
目录结构不一致
文档职责重复
```

应立即：

```txt
1. 承认问题
2. 重新生成文件
3. 明确验证关键内容存在
4. 如有同名缓存问题，生成带版本号的新文件
5. 给出新版下载链接
```

不要辩解，也不要说“应该有”。

---

## 18. 不建议继续扩张的情况

当文档体系已经覆盖以下内容时，不建议继续增加新规范：

```txt
项目定位
技术执行
视觉风格
内容管理
模块架构
组件设计
文案风格
部署运维
验收测试
Agent 指令
Agent 工作流
视觉参考
技术参考
后续功能
数据模型
阶段里程碑
隐私边界
```

后续新规则优先补入已有文档，而不是继续新增文件。

## 19. 推荐最终文档体系

对于个人网站 / 小型产品原型 / 静态项目，推荐最终控制在：

```txt
00–17
```

其中：

```txt
00–01：总入口
02–09：核心规范
10–11：Agent 执行
12–14：参考和规划
15–17：关键补充
```

这套体系已经足够支撑：

```txt
项目初始化
设计约束
内容管理
Agent 开发
阶段验收
部署上线
公开风险控制
```

---

## 20. 最终原则

> 先架构，后规范；先总览，后细则；先边界，后开发。  
> 规范文档不是越多越好，而是每份文档都有明确职责、主定义归属和执行价值。  
> Agent 不应自由发挥，而应按阶段、按规范、按验收标准小步推进。
