# 14_后续功能规划池

版本：v1.0  
项目：Pixel Archive  
位置：`docs/references/14_后续功能规划池.md`  
用途：记录 Pixel Archive 后续可能扩展的功能、页面、交互、内容系统和部署增强。本文档是规划池，不代表当前阶段要实现。  

---

## 1. 本文档的作用

本文档负责记录：

```txt
后续可能做什么
当前为什么不做
什么条件下可以做
做之前需要读哪些规范
可能影响哪些模块
```

本文档不是任务清单，也不是当前阶段开发计划。

当前阶段开发边界以：

```txt
docs/00_项目开发计划书.md
docs/standards/02_技术与开发执行规范.md
```

为准。

---

## 2. 使用原则

规划池中的内容只表示“以后可能考虑”。

Agent 不得因为本文档出现某功能，就主动实现。

任何功能从规划池进入开发，都必须满足：

```txt
用户明确要求
当前阶段允许
需求边界清楚
依赖影响已评估
涉及规范已阅读
验收标准已确定
```

---

## 3. 功能状态定义

规划池中的功能使用以下状态：

| 状态 | 含义 |
|---|---|
| idea | 只是想法 |
| candidate | 可能会做 |
| planned | 已决定后续做，但未进入当前阶段 |
| blocked | 暂不适合做 |
| deferred | 延后，不在近期做 |
| done | 已实现 |

默认状态为：

```txt
idea
```

---

## 4. 优先级定义

| 优先级 | 含义 |
|---|---|
| P0 | 当前阶段必须做 |
| P1 | 近期较重要 |
| P2 | 中期增强 |
| P3 | 长期想法 |
| P4 | 低优先级或实验性 |

本文档中大多数功能应为：

```txt
P2 / P3 / P4
```

---

## 5. 当前阶段不做的功能

第一阶段明确不做：

```txt
搜索
评论
CMS
数据库
后端
复杂像素头像
复杂像素房间
地图式首页
GitHub API 自动拉取
多语言
正式 SEO 深度优化
Open Graph 图片系统
复杂动画
访问统计
联系表单后端
PWA
```

这些功能可以放入规划池，但不得进入第一阶段开发。

## 32. 功能进入开发前检查

任何规划池功能进入开发前，必须确认：

```txt
1. 用户明确要求
2. 当前阶段允许
3. 是否需要新依赖
4. 是否影响技术栈
5. 是否影响目录结构
6. 是否影响内容模型
7. 是否影响组件接口
8. 是否影响部署
9. 是否需要新增规范
10. 是否有验收标准
```

## 34. Agent 使用规划池规则

Agent 必须遵守：

```txt
1. 本文档不是任务清单
2. 不得主动实现规划池功能
3. 不得因为本文档提到某技术就安装依赖
4. 不得用规划池覆盖当前阶段范围
5. 如用户要求实现规划池功能，应先检查阶段、依赖、规范和验收标准
```

## 36. 最终原则

> 后续功能规划池用于保存想法，不用于扩大当前阶段范围。  
> Pixel Archive 应先保持轻量、稳定、清晰，再按需要逐步加入搜索、RSS、SEO、像素头像、Lab 内容和其他增强功能。
