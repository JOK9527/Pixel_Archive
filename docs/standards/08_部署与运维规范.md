# 08_部署与运维规范

版本：v1.0  
项目：Pixel Archive  
位置：`docs/standards/08_部署与运维规范.md`


## 2. 部署总原则

Pixel Archive 初版是静态站点。

部署原则：

```txt
静态优先
构建简单
依赖克制
无后端
无数据库
无 CMS
本地先跑通
GitHub 管理版本
Cloudflare Pages 部署
域名后续再绑定
```

第一阶段目标不是部署复杂系统，而是保证：

```txt
本地可运行
构建可通过
仓库结构清楚
部署路径明确
后续上线成本低
```

---

## 3. 当前部署路线

推荐路线：

```txt
Phase 1：本地开发
Phase 2：GitHub 仓库管理
Phase 3：Cloudflare Pages 静态部署
Phase 4：基础域名绑定
Phase 5：后续 SEO / RSS / 搜索增强
```

当前第一阶段只需要重点保证：

```txt
npm run dev 可运行
npm run build 可通过
项目可以推送到 GitHub
Cloudflare Pages 后续可直接识别构建命令
```

## 4. 本地运行环境

## 4.1 基础要求

本地需要：

```txt
Node.js
npm
Git
代码编辑器
浏览器
```

建议：

```txt
Node.js 使用当前 LTS 版本
npm 使用 Node 自带版本即可
Git 使用稳定版本
```

不强制使用：

```txt
pnpm
yarn
bun
Docker
WSL
后端运行环境
数据库
```

除非后续明确切换。

---

## 4.2 本地安装

首次克隆或初始化后运行：

```bash
npm install
```

如果依赖异常，可尝试：

```bash
rm -rf node_modules
rm package-lock.json
npm install
```

Windows PowerShell 下可使用：

```powershell
Remove-Item -Recurse -Force node_modules
Remove-Item package-lock.json
npm install
```

但删除 lock 文件应谨慎，除非确实需要重装依赖。

---

## 4.3 本地开发

运行：

```bash
npm run dev
```

Astro 默认会启动本地开发服务器。

常见地址可能是：

```txt
http://localhost:4321
```

实际端口以终端输出为准。

不要在业务代码中写死本地端口。

---

## 4.4 本地构建

运行：

```bash
npm run build
```

构建成功后会生成静态输出目录，通常是：

```txt
dist/
```

`dist/` 是构建产物，不应手动修改。

---

## 4.5 本地预览

如果 package.json 中提供 preview 命令，可运行：

```bash
npm run preview
```

用于检查构建后的静态产物。

---

## 5. package.json 脚本规范

建议至少包含：

```json
{
  "scripts": {
    "dev": "astro dev",
    "build": "astro build",
    "preview": "astro preview"
  }
}
```

如果后续配置检查命令，可增加：

```json
{
  "scripts": {
    "check": "astro check"
  }
}
```

不建议第一阶段加入复杂脚本，例如：

```txt
复杂部署脚本
自定义构建管线
多环境脚本
自动压缩图片脚本
复杂 CI/CD 脚本
```

---

## 6. 端口规则

当前项目没有后端服务，也没有 API 服务。

因此：

```txt
不需要前后端端口对齐
不需要 API_BASE_URL
不需要 CORS 配置
不需要本地代理
```

Astro dev server 的端口只用于本地预览。

如果默认端口被占用，可以临时指定：

```bash
npm run dev -- --port 4322
```

不要把端口写入业务逻辑。

---

## 7. 环境变量规则

第一阶段原则上不需要环境变量。

不需要：

```txt
API Key
数据库地址
CMS token
后端 base URL
GitHub token
```

如果后续引入：

```txt
GitHub API 自动拉取
评论系统
联系表单
搜索服务
统计分析
CMS
```

才需要设计环境变量。

后续环境变量命名建议：

```txt
PUBLIC_SITE_URL
PUBLIC_GITHUB_URL
CONTACT_FORM_ENDPOINT
GITHUB_TOKEN
```

注意：

```txt
以 PUBLIC_ 开头的变量会暴露给前端
敏感 token 不能使用 PUBLIC_ 前缀
```

当前阶段不要提前添加无用 `.env`。

## 8. GitHub 仓库规范

## 8.1 仓库用途

GitHub 用于：

```txt
保存源码
记录版本
配合 Cloudflare Pages 部署
展示项目链接
```

不用于：

```txt
存储大量图片原始素材
存储隐私信息
存储 API Key
存储无关压缩包
```

---

## 8.2 .gitignore 建议

应忽略：

```txt
node_modules/
dist/
.astro/
.env
.env.local
.DS_Store
```

建议 `.gitignore` 包含：

```gitignore
node_modules/
dist/
.astro/

.env
.env.local
.env.*.local

.DS_Store
Thumbs.db
```

不要忽略：

```txt
src/content/
public/images/
docs/
README.md
WORKLOG.md
package-lock.json
```

`package-lock.json` 应保留，以保证依赖版本稳定。

---

## 8.3 Commit 规则

阶段验收通过后再 commit。

建议 commit message：

```txt
初始化 Pixel Archive 项目骨架
实现基础视觉系统与布局
实现首页档案终端初版
实现 Content Collections 与占位内容
完善部署配置与 README
```

不建议：

```txt
update
fix
test
final
随便改
```

---

## 8.4 Push 规则

默认：

```txt
git push 由用户手动执行
Agent 只提供 commit / push 指令
```

示例：

```bash
git status
git add .
git commit -m "初始化 Pixel Archive 项目骨架"
git push origin main
```

除非用户明确要求，不要让 Agent 自动 push。

## 9. Cloudflare Pages 部署规范

## 9.1 部署方式

推荐使用 Cloudflare Pages 连接 GitHub 仓库。

基本流程：

```txt
1. 将项目推送到 GitHub
2. 登录 Cloudflare
3. 创建 Pages 项目
4. 选择 GitHub 仓库
5. 设置构建命令
6. 设置输出目录
7. 部署
```

---

## 9.2 Cloudflare Pages 构建配置

建议配置：

```txt
Framework preset: Astro
Build command: npm run build
Build output directory: dist
Root directory: /
```

如果项目在仓库子目录中，再设置 Root directory。

---

## 9.3 Node 版本

Cloudflare Pages 可能需要指定 Node 版本。

建议后续如遇构建问题，再添加环境变量：

```txt
NODE_VERSION=20
```

或使用 Cloudflare 当前推荐 LTS 版本。

不要提前加入复杂版本控制，除非构建失败。

---

## 9.4 部署分支

默认部署分支：

```txt
main
```

如果使用其他分支，例如：

```txt
master
dev
preview
```

必须在 README 中说明。

初版建议只使用：

```txt
main
```

```txt

```


## 12. 构建产物规则

Astro 构建产物通常位于：

```txt
dist/
```

规则：

```txt
dist/ 不提交 Git
dist/ 不手动修改
dist/ 由 npm run build 生成
部署平台读取 dist/
```

如果发现 dist 被提交，应移除并更新 `.gitignore`。

静态资源放在：

```txt

```

内容图片路径规则见：


## 15. 部署后检查清单

Cloudflare Pages 部署完成后检查：

```txt
1. 线上首页正常打开
2. 导航链接正常
3. 页面刷新不 404
4. 暗色模式正常
5. 图片加载正常
6. CSS 样式正常
7. 404 页面正常
8. 移动端基础布局不崩
9. 控制台无明显资源 404
10. Cloudflare Pages 构建日志无错误
```

如发现问题，应先判断是：

```txt
构建问题
路径问题
资源问题
路由问题
主题脚本问题
部署配置问题
```

## 16. 常见部署问题

## 16.1 构建失败

可能原因：

```txt
依赖未安装
Node 版本不兼容
TypeScript 错误
Content Collections schema 错误
MDX 语法错误
路径大小写错误
```

处理：

```txt
查看 Cloudflare 构建日志
本地运行 npm run build 复现
先修本地构建
再重新部署
```

---

## 16.2 图片线上不显示

可能原因：

```txt
图片不在 public/images/
路径大小写错误
Markdown 使用了本地路径
Obsidian 图片语法未转换
图片文件未提交 Git
```

处理：

```txt
检查 public/images/
检查 Markdown 图片路径
检查 Git status
检查线上控制台 404
```

---

## 16.3 页面刷新 404

Astro 静态页面通常不应出现 SPA 刷新 404 问题。

如果出现，检查：

```txt
页面是否实际生成
路由是否正确
文件名是否符合 Astro 路由规则
Cloudflare 输出目录是否是 dist
```

---

## 16.4 样式线上异常

可能原因：

```txt
CSS 未正确引入
构建输出异常
缓存问题
路径问题
主题脚本报错
```

处理：

```txt
本地 npm run build + npm run preview
清理浏览器缓存
检查 Cloudflare 部署日志
检查控制台错误
```

---

## 16.5 暗色模式异常

可能原因：

```txt
data-theme 未正确设置
localStorage 逻辑错误
脚本执行时机错误
CSS 变量缺失
```

处理：

```txt
检查 html 上的 data-theme
检查 theme.css
检查 ThemeToggle
检查控制台报错
```

## 30. 最终原则

> Pixel Archive 的部署应保持简单、静态、可复现。  
> 先保证本地构建稳定，再接入 GitHub 和 Cloudflare Pages；不要过早引入复杂 CI/CD、后端、数据库或运行时服务。
