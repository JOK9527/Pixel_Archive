# 04_内容与文件管理规范

版本：v1.0  
项目：Pixel Archive  
位置：`docs/standards/04_内容与文件管理规范.md`  
用途：规范 Pixel Archive 中 Projects、Notes、Archive、Lab 等内容文件的命名、目录、slug、frontmatter、图片资源、标签分类、内容关联和 Obsidian 迁移规则。  

---

## 1. 本文档的作用

本文档负责回答：

```txt
1. 内容文件放在哪里？
2. Projects / Notes / Archive / Lab 如何命名？
3. slug 和 URL 如何保持稳定？
4. frontmatter 字段如何写？
5. 图片资源放在哪里、怎么引用？
6. category / tags 如何管理？
7. Obsidian 内容如何迁移？
8. 项目、文章、存档点之间如何关联？
```

本文档不负责定义：

```txt
技术栈和依赖管理
视觉风格
模块依赖方向
组件 props
文案语气
部署流程
验收命令
```

这些由其他专项规范负责。

---

## 2. 内容管理总原则

Pixel Archive 是长期维护的个人档案馆，内容系统必须稳定、清晰、可迁移。

核心原则：

```txt
内容结构清晰
文件命名稳定
图片路径可预测
frontmatter 字段统一
slug 长期不乱改
分类克制
标签规范
占位内容也按正式规范写
Obsidian 内容后续可迁移
```

不允许把内容目录做成大杂烩。

所有内容必须归入以下四类之一：

```txt
projects
notes
archive
lab
```

---

## 3. Content Collections 结构

Astro Content Collections 目录固定为：

```txt
src/content/
├─ projects/
├─ notes/
├─ archive/
└─ lab/
```

对应关系：

| Collection | 用途 |
|---|---|
| projects | 项目与作品 |
| notes | 文章、日志、学习笔记 |
| archive | 存档点、里程碑、阶段记录 |
| lab | 实验性内容、像素/UI/Minecraft/图像实验 |

不要新增以下目录作为并列内容入口，除非后续重新规划：

```txt
blog
posts
articles
works
portfolio
logs
gallery
```

当前统一使用：

```txt
Notes
```

作为文章与笔记入口。

## 5. 文件命名总规则

所有内容文件名必须使用：

```txt
小写英文
短横线连接
避免空格
避免中文文件名
避免特殊符号
避免过长
```

推荐：

```txt
scholar-lens-ai.mdx
servo-test-platform.mdx
2026-06-10-first-note.mdx
pixel-ui-placeholder.mdx
```

不推荐：

```txt
ScholarLens AI.md
我的第一篇文章.md
2026/06/10 note.md
project_final_version_修改版.md
论文阅读（最终）.md
test.md
demo.md
aaa.md
```

## 6. Projects 文件命名

Projects 文件名使用项目 slug：

```txt
项目英文名 / 简短英文描述
```

示例：

```txt
pixel-archive.mdx
scholar-lens-ai.mdx
servo-test-platform.mdx
glm-ocr-tool.mdx
flight-control-learning-log.mdx
```

规则：

```txt
不加日期前缀
文件名应长期稳定
项目改名时谨慎修改 slug
项目标题可以在 frontmatter 中使用正式写法
```

示例：

```txt
文件名：scholar-lens-ai.mdx
标题：ScholarLens AI
URL：/projects/scholar-lens-ai/
```

---

## 7. Notes 文件命名

Notes 文件建议使用日期前缀：

```txt
YYYY-MM-DD-title-slug.md
YYYY-MM-DD-title-slug.mdx
```

示例：

```txt
2026-06-10-astro-content-collections.md
2026-06-10-personal-website-planning.mdx
2026-06-11-flight-control-learning-note.md
```

规则：

```txt
日期使用发布时间或写作开始日期
标题 slug 使用英文简写
同一天多篇文章可自然区分
普通 Note / Log 建议加日期
长期专题索引可不加日期
```

长期专题示例：

```txt
flight-control-index.md
machine-learning-paper-reading.md
```

---

## 8. Archive 文件命名

Archive 是存档点，不是普通文章。

文件名建议：

```txt
YYYY-MM-short-event.md
```

示例：

```txt
2026-06-started-pixel-archive.md
2026-05-graduation-project-wrap-up.md
2026-05-scholar-lens-vlm-parser.md
```

规则：

```txt
Archive 按月份归档
一条 archive 文件对应一个重要阶段
不记录过碎的日常小事
后续如数量增加，可按年份建子目录
```

---

## 9. Lab 文件命名

Lab 用于实验内容，文件名使用实验主题 slug。

示例：

```txt
pixel-ui-placeholder.mdx
minecraft-build-placeholder.mdx
image-generation-experiment.mdx
terminal-widget-experiment.mdx
```

规则：

```txt
不强制日期
一次性实验可以加日期
持续维护实验不加日期更合适
```

示例：

```txt
2026-06-10-pixel-button-test.mdx
pixel-button-system.mdx
```

---

## 10. Slug 规则

slug 是 URL 和内容关联的核心，应保持稳定。

slug 使用：

```txt
小写英文
短横线连接
不使用中文
不使用空格
不使用下划线
不使用特殊符号
```

推荐：

```txt
scholar-lens-ai
servo-test-platform
astro-content-collections
pixel-ui-placeholder
```

不推荐：

```txt
ScholarLensAI
scholar_lens_ai
我的项目
astro content collections
project#1
```

默认情况下：

```txt
文件名 = slug
```

Notes 如果有日期前缀，第一阶段可以直接使用完整文件名作为 slug。

---

## 11. URL 规则

初版 URL 建议：

```txt
/
/projects/
/projects/{slug}/
/notes/
/notes/{slug}/
/archive/
/lab/
/about/
```

示例：

```txt
/projects/pixel-archive/
/projects/scholar-lens-ai/
/notes/2026-06-10-astro-content-collections/
/lab/pixel-ui-placeholder/
```

不要混用：

```txt
/blog/
/articles/
/posts/
```

当前统一使用：

```txt
/notes/
```

---

## 12. Slug 修改规则

上线后不要随意修改 slug。

如果必须修改：

```txt
1. 检查是否已有外链
2. 检查是否被 Archive / Notes / Projects 引用
3. 更新相关 related 字段
4. 必要时添加 redirect
5. 更新 README / WORKLOG 中相关说明
```

不允许为了“看起来更顺眼”频繁修改 slug。

---

## 13. Frontmatter 总规则

所有内容文件必须有 frontmatter。

日期统一使用：

```txt
YYYY-MM-DD
```

数组统一使用 YAML 数组：

```yaml
tags:
  - astro
  - markdown
  - personal-site
```

不要写成：

```yaml
tags: astro, markdown, personal-site
```

布尔值使用：

```yaml
draft: true
featured: false
```

字符串建议加引号，尤其包含符号、斜线或中文时。

## 15. Project 字段取值

## 15.1 status

Project status 只允许：

```txt
WIP
Done
Prototype
Paused
```

含义：

| Status | 含义 |
|---|---|
| WIP | 正在开发或持续维护 |
| Done | 阶段性完成 |
| Prototype | 原型 / 试验项目 |
| Paused | 暂停维护 |

不要使用：

```txt
doing
finished
complete
todo
in-progress
```

---

## 15.2 category

初版 Project category 建议固定为：

```txt
AI / ML
Aerospace / Control
Tools / Web
Creative / Lab
```

规则：

```txt
category 数量要少
不要频繁新增 category
更细关键词放 tags
```

---

## 15.3 tags

Project tags 规则：

```txt
使用小写英文
不使用空格
多词用短横线
不要滥用标签
每个项目建议 3–8 个 tag
```

推荐：

```txt
pdf
ai
vlm
markdown
fastapi
astro
control
matlab
```

## 17.1 type

Note type 只允许：

```txt
Article
Log
```

含义：

| Type | 含义 |
|---|---|
| Article | 正式文章，结构完整，可长期公开 |
| Log | 开发日志 / 阶段记录 |
| Note | 学习笔记 / 临时整理 / 知识卡片 |

```txt
Web
AI / ML
Aerospace / Control
```

规则：

```txt

一篇 Note 只有一个 category
更细关键词放 tags
```

```txt
使用小写英文
多词用短横线
每篇文章建议 3–10 个 tag
不要和 category 完全重复太多
```

示例：

```txt

markdown
typescript
flight-control
servo
```

## 19. Archive 字段取值

Archive type 只允许：

```txt
save-point
milestone
project-log
```

含义：

| Type | 含义 |
|---|---|
| save-point | 普通存档点 |
| milestone | 重要里程碑 |
| project-log | 项目阶段记录 |

Archive 不是日记，不记录太碎内容。

适合记录：

```txt
新项目启动
项目阶段完成
重要学习阶段
网站上线
论文 / 毕设 / 工具开发阶段性完成
```

不适合记录：

```txt
无长期意义的琐事
过细 bug 修复
临时想法
日常碎碎念
```

## 24. 图片引用规则

Markdown / MDX 中统一使用 public 绝对路径：

```md
![ScholarLens AI screenshot](/images/projects/scholar-lens-ai/screenshot-01.png)
```

不要使用：

```md
![](./assets/screenshot.png)
![](C:/Users/xxx/Desktop/image.png)
![[Obsidian Image.png]]
```

所有正文图片都应写 alt 文本：

```md
![Archive Terminal layout sketch](/images/notes/pixel-archive/terminal-layout.png)
```

不推荐：

```md
![](/images/notes/pixel-archive/terminal-layout.png)
```

## 26. Category 与 Tag 总规范

## 26.1 Category

category 是主分类，数量要少。

规则：

```txt
一篇内容只有一个 category
category 可以使用英文短语
不要把 category 当 tag 用
不要频繁新增 category
```

---

## 26.2 Tag

tag 是关键词，数量可以稍多。

规则：

```txt
使用小写英文
多词用短横线
不要使用中文
不要随意造近义 tag
```

推荐统一：

```txt
machine-learning
paper-reading
flight-control
system-identification
astro
typescript
markdown
pixel-ui
web
vlm
pdf
```

不要同时出现：

```txt
ml
machine-learning
machinelearning
MachineLearning
```

应统一用一个。

---

## 27. Draft 草稿规则

所有 collection 后续都可以扩展 draft 字段，但第一阶段主要用于 Notes。

```yaml
draft: true
```

表示：

```txt
本地可以显示或隐藏
部署时默认不展示 draft
draft 内容可以不完整
```

正式发布：

```yaml
draft: false
```

或省略 draft，但建议明确写出。

## 28. 内容关联规则

Project 可以关联 Notes：

```yaml
relatedNotes:
  - "2026-06-10-scholar-lens-design-note"
```

Note 可以关联 Projects：

```yaml
relatedProjects:
  - "scholar-lens-ai"
```

Archive 可以关联 Project 或 Note：

```yaml
relatedProject: "pixel-archive"
relatedNote: "2026-06-10-personal-website-planning"
```

关联字段必须使用对应内容的 slug，不使用标题。

推荐：

```yaml
relatedProject: "pixel-archive"
```

不推荐：

```yaml
relatedProject: "Pixel Archive 个人网站"
```

## 29. Obsidian 迁移规则

用户后续可能在 Obsidian 中写内容，再迁移到 Pixel Archive。

建议 Obsidian 中也尽量使用相同 frontmatter。

---

## 29.1 Obsidian 文件名

Obsidian 内部可以用中文标题，但迁移到网站时必须改为 slug 文件名。

示例：

```txt
Obsidian: Astro 内容集合整理.md
网站: 2026-06-10-astro-content-collections.md
```

---

## 29.2 Obsidian 图片

Obsidian 图片通常可能是：

```md
![[image.png]]
```

迁移到网站时必须改成标准 Markdown：

```md
![Image description](/images/notes/2026-06-10-astro-content-collections/image.png)
```

---

## 29.3 Obsidian 双链

Obsidian 双链：

```md
[[ScholarLens AI]]
```

迁移到网站时应改为标准链接：

```md
[ScholarLens AI](/projects/scholar-lens-ai/)
```

---

## 29.4 Obsidian 标签

Obsidian 标签：

```txt
#machine-learning
```

迁移到网站时应放入 frontmatter tags，不建议在正文里保留大量裸标签。

---

## 30. 内容写作语言规则

初版内容以中文为主。

页面标题和导航可以使用英文：

```txt
Home
Projects
Notes
Archive
Lab
About
```

正文可以中文为主，保留必要英文技术术语。

项目名可使用英文：

```txt
Pixel Archive
ScholarLens AI
GLM-OCR Tool
Servo Test Platform
```

不要强行全文英文，除非后续明确要做双语。

具体文案风格由：

```txt
07_文案与内容风格规范.md
```

定义。

---

## 31. 空状态内容规则

页面内容还少时，不能显得像报错。

空状态文案应保持 Pixel Archive 风格。

示例：

```txt
No files loaded yet.
This drawer is still empty.
New archive slot waiting.
Nothing here, for now.
```

中文页面也可以用短句：

```txt
这里还没有内容。
新的存档槽等待写入。
```

避免：

```txt
暂无数据
加载失败
列表为空
```

除非是真的错误状态。

## 32. 示例占位内容清单

第一阶段建议创建：

## 34. 内容验收标准

检查内容系统时，至少确认：

```txt
1. src/content 下只有 projects / notes / archive / lab 四类内容目录
2. 文件名均为小写英文 + 短横线
3. Notes 文件含日期前缀
4. Projects 文件不含日期前缀
5. 所有内容都有 frontmatter
6. frontmatter 字段符合 schema
7. status / type 使用固定枚举
8. tags 使用小写英文
9. 图片路径使用 /images/... 绝对路径
10. 没有中文图片文件名
11. 没有 Obsidian 双链残留
12. 没有本地绝对路径图片引用
13. 占位内容也符合正式规范
14. Project / Note / Archive 关联字段使用 slug，不使用标题
15. 内容少时页面空状态仍然自然
```

完整阶段验收由：

```txt
09_验收与测试规范.md
```

定义。

> Pixel Archive 的内容系统应当像一个长期可维护的档案库：文件名稳定，frontmatter 统一，图片路径清晰，标签分类克制，所有内容都能被 Projects、Notes、Archive 和 Lab 四个入口自然组织起来。
