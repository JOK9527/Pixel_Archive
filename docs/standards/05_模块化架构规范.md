# 05_模块化架构规范

版本：v1.0  
项目：Pixel Archive  
位置：`docs/standards/05_模块化架构规范.md`  
用途：规范 Pixel Archive 的模块划分、模块职责、依赖方向、数据流、内部接口、目录边界和禁止耦合方式，确保项目长期可维护。  

---

## 1. 本文档的作用

本文档负责回答：

```txt
1. Pixel Archive 应该拆成哪些模块？
2. 每个模块负责什么？
3. 页面、布局、业务组件、UI 组件、内容读取、样式、配置之间如何分工？
4. 模块之间允许怎样依赖？
5. 模块之间不允许怎样依赖？
6. 数据应该从哪里读取、如何传递到组件？
7. src/lib/content 是否需要，负责什么？
8. 如何避免后期变成一个混乱的大文件项目？
```

本文档不负责定义：

```txt
技术栈和依赖管理
视觉色彩和字体
内容文件命名
具体组件 props
文案风格
部署流程
验收命令
```

这些分别由其他专项规范负责。

## 2. 模块化总原则

Pixel Archive 是一个静态个人网站，但仍然需要模块化。

模块化目标不是过度工程化，而是保证：

```txt
目录清楚
职责清楚
依赖清楚
数据流清楚
组件可复用
样式不污染
内容系统稳定
后续功能可扩展
```

核心原则：

> 页面负责组织数据，业务组件负责展示业务内容，UI 组件负责通用视觉，内容层负责读取和整理数据，样式层负责统一主题变量。

## 3. 推荐整体架构

推荐架构分层：

```txt
pages
  ↓
layouts
  ↓
business components
  ↓
ui components
  ↓
styles / tokens
```

内容数据流：

```txt
src/content
  ↓
src/lib/content
  ↓
pages
  ↓
business components
  ↓
ui components
```

配置数据流：

```txt
src/data
  ↓
layouts / pages / components
```

## 5. 模块划分总览

Pixel Archive 建议划分为以下模块：

| 模块 | 路径 | 职责 |
|---|---|---|
| layout 模块 | `components/layout/` + `layouts/` | 页面外壳、导航、页脚、主题切换 |
| home 模块 | `components/home/` | 首页 Archive Terminal、Save Slots、Status Bar |
| project 模块 | `components/project/` | 项目列表、项目卡片、项目详情展示 |
| notes 模块 | `components/notes/` | Notes 列表、正文展示、TOC、文章元信息 |
| archive 模块 | `components/archive/` | 时间线、存档点、年份 / 月份分组 |
| lab 模块 | `components/lab/` | Lab 卡片、实验内容网格 |
| ui 模块 | `components/ui/` | Button、Card、Tag、Badge、EmptyState 等通用组件 |
| content 模块 | `content/` + `lib/content/` | 内容文件和内容读取整理 |
| styles 模块 | `styles/` | 主题变量、全局样式、Markdown 样式 |
| data 模块 | `data/` | 站点配置、导航配置、静态常量 |

## 6. 模块依赖方向

必须保持单向依赖。

推荐依赖方向：

```txt
pages
  ↓
layouts / business components
  ↓
ui components
  ↓
styles / tokens
```

内容读取方向：

```txt
content collections
  ↓
lib/content
  ↓
pages
  ↓
components
```

配置读取方向：

```txt
data
  ↓
layouts / pages / components
```

---

## 7. 允许的依赖关系

## 7.1 pages 可以依赖

```txt
layouts
components
lib/content
data
styles indirectly through layouts
```

示例：

```txt
pages/projects/index.astro
  可以调用 ProjectGrid
  可以调用 getAllProjects()
  可以使用 BaseLayout
```

---

## 7.2 layouts 可以依赖

```txt
components/layout
components/ui
data
styles
```

示例：

```txt
BaseLayout.astro
  可以使用 Header
  可以使用 Footer
  可以读取 site.ts
```

---

## 7.3 business components 可以依赖

业务组件包括：

```txt
components/home
components/project
components/notes
components/archive
components/lab
```

它们可以依赖：

```txt
components/ui
局部样式
必要的类型定义
```

示例：

```txt
ProjectCard 可以使用 Card / Badge / Tag
NoteCard 可以使用 Card / Tag
ArchiveTimeline 可以使用 StatusPill
```

---

## 7.4 ui components 可以依赖

UI 组件只能依赖：

```txt
CSS Variables
自身 props
极少量通用类型
```

UI 组件不应依赖：

```txt
projects
notes
archive
lab
site-specific business logic
Content Collections
pages
```

---

## 7.5 lib/content 可以依赖

```txt
astro:content
内容 schema 类型
纯数据整理函数
```

不应依赖：

```txt
组件
页面
样式
浏览器 API
```

---

## 7.6 data 可以依赖

`src/data/` 应尽量是纯静态配置。

可以包含：

```txt
site title
description
navigation
social links
footer quote list
homepage menu items
```

不应依赖：

```txt
components
pages
content collections
浏览器 API
```

---

## 8. 禁止的反向依赖

以下依赖方式不允许：

```txt
ui 组件依赖业务组件
ui 组件读取 Content Collections
业务组件直接读取所有内容文件
Header 在组件内部随意写死导航
ProjectCard 自己 getCollection("projects")
NoteCard 自己 getCollection("notes")
ArchiveTerminal 自己读取 projects collection
styles 依赖组件结构
data 依赖组件
content 依赖页面
```

具体例子：

```txt
不推荐：
ProjectCard.astro 内部调用 getCollection("projects")

推荐：
pages/projects/index.astro 调用 getAllProjects()
然后把单个 project 传给 ProjectCard
```

## 9. 页面层规范

`src/pages/` 负责路由和页面组装。

页面层职责：

```txt
定义 URL 路由
调用 layout
读取内容数据
组织页面结构
把数据传给组件
处理页面级 metadata
```

页面层不应承担：

```txt
复杂视觉组件实现
大量业务卡片 HTML
重复内容筛选逻辑
大型 CSS
复杂数据整理逻辑
```

如果页面文件过长，应该拆分到组件或 `lib/content/`。

## 11. Business Components 规范

业务组件负责展示具体业务内容，但不负责全局数据读取。

业务组件目录：

```txt
components/home/
components/project/
components/notes/
components/archive/
components/lab/
```

业务组件应：

```txt
接收 props
展示数据
组合 UI 组件
保持自身职责单一
不直接操作全局状态
不擅自读取 collection
```

路径：

```txt
components/ui/
```

```txt
Button.astro
Card.astro
Tag.astro
Badge.astro
StatusPill.astro
EmptyState.astro
IconLink.astro
SectionHeader.astro
```

UI 组件职责：

```txt
提供通用视觉构件
不包含业务含义
可被多个业务模块复用

```

```txt

```


## 18. Content 模块

Content 模块由两部分组成：

```txt
src/content/
src/lib/content/
```

## 18.1 src/content/

职责：

```txt
保存 Markdown / MDX 内容文件
由 Astro Content Collections 管理
由 src/content/config.ts 定义 schema
```

不应包含：

```txt
组件代码
样式代码
页面代码
复杂数据处理逻辑
```

内容命名规则见：

```txt
04_内容与文件管理规范.md
```

---

## 18.2 src/lib/content/

职责：

```txt
封装 getCollection 读取逻辑
筛选 draft
排序内容
按 featured 筛选
按 category / tag 分组
生成页面所需的简化数据
统一 slug / href 生成逻辑
```

推荐文件：

```txt
src/lib/content/projects.ts
src/lib/content/notes.ts
src/lib/content/archive.ts
src/lib/content/lab.ts
```

示例函数：

```txt
getAllProjects()
getFeaturedProjects()
getProjectBySlug()
getAllNotes()
getPublishedNotes()
getNotesByType()
getArchiveGroups()
getAllLabItems()
```

第一阶段如果内容很少，可以先在页面中直接读取 collection。  
但一旦两个以上页面重复同类读取逻辑，应抽到 `src/lib/content/`。

## 20. Styles 模块

路径：

```txt
src/styles/
```

文件职责：

```txt
theme.css：主题变量、颜色、字体、阴影、圆角等 tokens
global.css：reset、body、基础布局、链接、滚动条等全局样式
markdown.css：Markdown 正文排版
```

规则：

```txt
组件局部样式写在组件内
全局样式只写真正全局的内容
不要把 ProjectCard 样式写进 global.css
不要让 markdown.css 影响首页和项目卡片
不要在组件里大量硬编码颜色
```

视觉细节由：

```txt
03_视觉风格规范.md
```

定义。

## 21. 内部接口原则

当前阶段没有后端 API，但有内部接口。

内部接口包括：

```txt
Content Collections schema
页面路由
组件 props
数据读取函数
CSS Variables
导航数据结构
图片路径
slug
```

这些接口应保持稳定。

不允许随意修改：

```txt
schema 字段名
组件 props 名称
路由路径
CSS 变量名
content slug
navigation item 结构
```

如果必须修改，应：

```txt
说明原因
说明影响范围
更新相关调用处
更新对应规范
更新 WORKLOG
```

## 22. 数据流规范

推荐数据流：

```txt
Content files
  ↓
Astro Content Collections
  ↓
src/lib/content/*
  ↓
pages
  ↓
business components
  ↓
ui components
```

配置数据流：

```txt
src/data/*
  ↓
layouts / pages
  ↓
components
```

不推荐：

```txt
ui components 直接读 content
business components 到处 getCollection
多个页面重复写复杂筛选逻辑
组件内部拼接不稳定 URL
```

---

## 23. 页面与组件通信

页面向组件传递数据。

推荐：

```txt
page 读取 projects
page 生成 href
page 传给 ProjectGrid
ProjectGrid 传给 ProjectCard
ProjectCard 使用 UI 组件展示
```

不推荐：

```txt
ProjectCard 自己判断 URL
ProjectCard 自己读取项目列表
ProjectGrid 自己读取全局配置并改变导航
```

组件应尽量是：

```txt
输入明确
输出 HTML
无隐藏副作用
```

---

## 24. 模块内部组织规则

每个模块内部也要保持职责清楚。

例如 Project 模块：

```txt
components/project/
├─ ProjectCard.astro
├─ ProjectGrid.astro
├─ ProjectMeta.astro
├─ ProjectLinks.astro
└─ FeaturedProjects.astro
```

不要把所有项目相关 HTML 都塞进：

```txt
pages/projects/index.astro
```

也不要建立：

```txt
ProjectEverything.astro
```

如果一个组件过长，应拆分。

---

## 25. 组件拆分时机

当出现以下情况时，应考虑拆分组件：

```txt
一个组件超过约 200–300 行
一个组件同时处理多个业务区域
同一段结构在多个页面重复出现
样式难以维护
props 变得混乱
组件名称无法准确描述职责
```

拆分原则：

```txt
先按职责拆
再按复用拆
不要为了拆而拆
不要制造过多空壳组件
```

---

## 26. 页面文件过长处理

如果页面文件过长，应判断原因：

```txt
HTML 结构太多 → 拆到业务组件
数据处理太多 → 拆到 lib/content
样式太多 → 放到组件 style 或 styles
配置太多 → 放到 data
```

页面文件应保持：

```txt
路由清晰
数据读取清晰
组件组合清晰
```

## 28. 禁止耦合方式

禁止以下耦合：

```txt
组件直接依赖页面路径结构
UI 组件写死业务文案
业务组件自己读取 collection
多个模块互相 import
样式文件依赖具体 DOM 深层结构
内容文件中写死本地绝对路径
导航在多个地方重复定义
同一数据筛选逻辑在多个页面复制
```

具体反例：

```txt
Button.astro 内部判断 href 是否包含 "/projects/"
ProjectCard.astro 内部 import getCollection
Header.astro 内部写死 nav items，同时 navigation.ts 也有一份
markdown.css 中写 .project-card h2
```

## 29. 可接受的轻量重复

模块化不是零重复。

以下轻量重复可以接受：

```txt
少量页面标题结构重复
少量 class 命名相似
第一阶段内容很少时页面直接 getCollection
简单空状态文案临时重复
```

但当重复开始影响维护时，应抽象到：

```txt
components
lib/content
data
```

## 30. 第一阶段模块化要求

第一阶段至少应完成：

```txt
components/layout/
components/home/
components/ui/
layouts/
pages/
styles/
content/
data/
```

可选但推荐预留：

```txt
components/project/
components/notes/
components/archive/
components/lab/
lib/content/
```

第一阶段不要求把所有模块做完整，但目录边界要清楚。

---

## 31. 后续扩展模块

后续如果加入新功能，应按模块化原则扩展。

可能新增：

```txt
components/search/
lib/search/
components/seo/
lib/seo/
components/rss/
lib/rss/
```

但新增前必须确认：

```txt
是否属于当前阶段
是否已有合适模块
是否会引入新依赖
是否需要更新规范
```

不要随意创建：

```txt
utils/
helpers/
common/
shared/
```

如果需要这些目录，必须先说明职责边界。

---

## 32. 与 Agent 的关系

Agent 开发时应先判断任务涉及哪些模块。

示例：

```txt
新增首页 Save Slot 动效
→ home 模块 + ui 模块 + 视觉规范

新增 ProjectCard
→ project 模块 + ui 模块 + 组件设计规范

新增 Notes 正文排版
→ notes 模块 + styles/markdown.css + 视觉规范

新增内容读取函数
→ lib/content 模块 + 内容管理规范
```

Agent 不应因为方便而跨模块乱写。

---

## 33. 模块化验收标准

检查模块化时，至少确认：

```txt
1. pages 主要负责路由和组合，不堆大量组件实现
2. layouts 只负责页面外壳，不写业务列表逻辑
3. business components 不直接读取 Content Collections
4. ui components 不包含业务含义
5. Header 导航来自 data/navigation.ts 或统一配置
6. 内容筛选逻辑没有在多个页面重复
7. 样式没有严重全局污染
8. markdown.css 不影响非 Markdown 页面
9. 组件命名能表达职责
10. 没有 test/demo/aaa 这类临时组件
11. 没有多个重复用途的 common/shared 目录
12. 模块依赖方向没有反转
13. 数据流清楚，props 传递明确
14. 新增模块有明确职责
15. 内部接口变更有记录
```

更完整验收见：

```txt
09_验收与测试规范.md
```

## 35. 最终原则

> Pixel Archive 的模块化不是为了复杂，而是为了长期干净。  
> 页面组织数据，业务组件展示内容，UI 组件提供通用视觉，内容层统一读取数据，样式层统一主题。  
> 任何模块都应该知道自己负责什么，也应该知道自己不该负责什么。
