# 09_验收与测试规范

版本：v1.0  
项目：Pixel Archive  
位置：`docs/standards/09_验收与测试规范.md`  
用途：规范 Pixel Archive 每轮开发、阶段完成、页面实现、视觉打磨、内容新增、部署上线前后的验收与测试标准。  

---

## 1. 本文档的作用

本文档负责回答：

```txt
1. 怎么判断一轮开发真的完成了？
2. 怎么判断一个阶段可以提交？
3. 怎么检查构建是否通过？
4. 怎么检查页面是否正常？
5. 怎么检查视觉是否符合 Pixel Archive 风格？
6. 怎么检查内容、文件、图片、frontmatter 是否规范？
7. 怎么检查模块化和组件边界？
8. 怎么检查暗色模式、移动端和基础可访问性？
9. Agent 完成任务后应该如何自检？
```

本文档不负责定义：

```txt
项目目标
技术栈
视觉规则本身
内容字段本身
模块架构本身
组件 props 本身
部署配置本身
```

这些由其他专项规范负责。本文只负责“怎么验收”。

---

## 2. 验收总原则

Pixel Archive 的验收目标不是追求复杂测试体系，而是确保每轮开发：

```txt
能运行
能构建
不破坏结构
不破坏视觉方向
不破坏内容规范
不引入未说明依赖
不产生明显耦合
不留下无法解释的问题
```

核心原则：

> 构建通过是最低标准；结构清晰、风格一致、内容规范、可继续开发才算真正完成。

---

## 3. 验收分层

验收分为五层：

```txt
基础运行验收
页面功能验收
规范一致性验收
视觉与体验验收
阶段完成验收
```

对应含义：

| 层级 | 重点 |
|---|---|
| 基础运行验收 | npm install / dev / build 是否正常 |
| 页面功能验收 | 页面是否存在、导航是否可用、路由是否正确 |
| 规范一致性验收 | 技术栈、内容、模块、组件是否遵守规范 |
| 视觉与体验验收 | 风格、暗色模式、移动端、可访问性 |
| 阶段完成验收 | README / WORKLOG / Git / 下一步是否明确 |

---

## 4. 每轮开发最低验收

每轮开发完成后，至少检查：

```txt
1. npm run build 通过
2. 本轮目标已完成
3. 修改文件范围可解释
4. 没有引入未说明依赖
5. 没有破坏已有页面
6. 没有违反当前阶段边界
7. README 或 WORKLOG 已按需更新
8. 已知问题已说明
9. 下一步建议明确
10. Git 提交建议明确
```

如果 `npm run build` 失败，不得声称本轮完成。

## 5. 命令验收

## 5.1 安装验收

首次或依赖变化后运行：

```bash
npm install
```

通过标准：

```txt
依赖安装完成
没有严重 error
package-lock.json 与 package.json 一致
```

如果失败，记录：

```txt
失败命令
错误信息
Node / npm 版本
是否新增依赖
已尝试处理方式
```

---

## 5.2 开发服务器验收

运行：

```bash
npm run dev
```

通过标准：

```txt
Astro dev server 正常启动
终端输出本地访问地址
浏览器可以打开首页
无明显启动错误
```

注意：

```txt
端口可能变化，以终端输出为准。
不要将端口写死到业务代码。
```

---

## 5.3 构建验收

运行：

```bash
npm run build
```

通过标准：

```txt
构建命令退出码为 0
dist/ 正常生成
没有 TypeScript / Astro / MDX / Content Collections 错误
没有未处理的路径错误
```

构建失败时必须说明：

```txt
失败命令
错误摘要
可能原因
已尝试修复
剩余问题
```

---

## 5.4 预览验收

如项目支持：

```bash
npm run preview
```

通过标准：

```txt
构建产物可预览
首页可访问
导航可用
主要页面可访问
```

## 6. 项目结构验收

检查目录是否符合规划：

```txt
src/
├─ components/
├─ content/
├─ layouts/
├─ pages/
├─ styles/
├─ data/
└─ lib/
```

第一阶段至少应存在：

```txt
src/components/
src/layouts/
src/pages/
src/styles/
src/content/
src/data/
```

推荐存在：

```txt
src/lib/content/
```

检查项：

```txt
1. 没有随意创建 test / demo / temp 目录
2. 没有重复用途的 common / shared / base 混乱目录
3. components 按模块分组
4. pages 只负责路由和页面组合
5. styles 文件职责清楚
6. content 下只有 projects / notes / archive / lab
7. public/images 用于图片资源
8. docs 规范文档结构清楚
```

检查：

```txt
1. 使用 Astro
2. 使用 TypeScript
3. 支持 MDX
4. 使用 Astro Content Collections
5. 使用原生 CSS + CSS Variables
6. 未引入 Tailwind
7. 未引入 shadcn/ui

10. 未引入数据库
11. 未引入后端服务
12. 新增依赖均有说明
```

如发现新增依赖，必须确认：

```txt
依赖名称
用途
新增原因
是否必要
是否属于当前阶段
```

## 8. 页面路由验收

初版页面应包含：

```txt
/
/projects/
/notes/
/archive/
/lab/
/about/
/404
```

检查项：

```txt
1. 首页可打开
2. Projects 页面可打开
3. Notes 页面可打开
4. Archive 页面可打开
5. Lab 页面可打开
6. About 页面可打开
7. 404 页面存在
8. 顶部导航链接正确
9. 页面标题和内容匹配
10. 路由没有混用 /blog、/articles、/posts
```

Projects 和 Notes 详情页后续应检查：

```txt
/projects/{slug}/
/notes/{slug}/
```

## 9. 首页验收

首页是 Pixel Archive 的核心入口。

检查项：

```txt
1. 显示 Pixel Archive
2. 首页不是传统个人简介页
3. 没有大段自我介绍
4. 没有真实姓名作为主视觉
5. 有 Archive Terminal 或同等入口组件
6. 有游戏菜单式入口
7. 有 Save Slots
8. 有 Status Bar
9. 文案短而克制
10. 有轻微像素 / 终端 / 存档感
11. 不依赖复杂大图素材
12. 不像普通博客首页
```

## 15. 404 页面验收

检查项：

```txt
1. 404 页面存在
2. 使用 lost archive / save file not found 概念

```

检查 `src/content/`：

检查：

```txt
1. 文件名小写英文
2. 使用短横线
3. 无空格
4. 无中文文件名
5. 无特殊符号
6. Projects 文件不加日期前缀

```

## 19. 图片资源验收

检查：

```txt
1. 图片放在 public/images/
2. 按 projects / notes / archive / lab / shared 分类
3. 图片文件名小写英文
4. 不使用中文图片文件名
5. Markdown 图片使用 /images/... 绝对路径
6. 没有 C:/Users/... 本地路径
7. 没有 Obsidian ![[image.png]] 残留

```

## 20. 模块化验收

检查：

```txt
1. pages 主要负责路由和组合
2. layouts 只负责页面外壳
3. business components 不直接读取 Content Collections
4. ui components 不包含业务含义
5. Header 导航来自统一配置
6. 内容读取逻辑没有大量重复
7. 样式没有严重全局污染
8. markdown.css 不影响非 Markdown 页面
9. 组件命名清楚
10. 没有临时组件和混乱目录
11. 模块依赖方向没有反转
12. 数据流清楚
13. 新增模块职责明确
```

---

## 21. 组件验收

检查：

```txt
1. 组件命名使用 PascalCase
2. UI 组件无业务含义
3. 业务组件职责单一
4. props 简洁明确
5. status / type 与内容规范一致
6. 组件样式使用 CSS Variables
7. 没有大量硬编码颜色
8. Button / Card / Badge / Tag 可复用
9. ProjectCard 与 NoteCard 边界清楚
10. Home 组件不承担 Projects 数据筛选
11. Notes 正文组件不被像素风破坏
12. 新增组件能说明所属模块和职责
```

---

## 22. 视觉验收

检查整体风格是否符合：

```txt
极简做骨架
像素做识别
复古做氛围
```

具体检查：

```txt
1. 首页有 Pixel Archive 独特气质
2. Projects 信息清楚，不杂乱
3. Notes 阅读舒适
4. Archive 有存档点感觉
5. Lab 更自由但不脱离整体
6. About 克制
7. 404 有趣但不模板化
8. 紫绿点缀低饱和
9. 没有高饱和刺眼颜色
10. 像素元素没有影响正文阅读
11. 第三方参考没有直接套模板感
12. 所有页面看起来属于同一个网站
```

## 23. 暗色模式验收

检查：

```txt
1. 主题切换按钮存在
2. 默认浅色
3. 可切换暗色
4. 暗色模式不是纯黑荧光绿终端
5. 暗色背景柔和
6. 正文可读
7. 卡片层级清楚
8. 链接和按钮可辨识
9. 代码块可读
10. 刷新后主题状态可保留或行为合理
11. 没有明显闪烁或错色
```

---

## 24. 移动端基础验收

虽然桌面端优先，但移动端不能崩。

检查：

```txt
1. 首页终端不横向溢出
2. 导航在窄屏可用
3. 项目卡片单列显示合理
4. Notes 正文宽度适配
5. 代码块可横向滚动
6. 表格可横向滚动
7. 图片不超出屏幕
8. Footer 不挤压
9. 按钮和链接可点击
10. 没有明显布局错位
```

第一阶段不要求完美移动端，但不能无法使用。

```txt
3. 链接可聚焦

```

## 26. 文案验收

检查：

```txt

3. Projects 文案说明项目，不营销

```

## 28. README 验收

README 至少包含：

```txt
项目简介
技术栈
本地运行方式
构建方式
目录结构简述
```

## 30. Agent 每轮验收流程

Agent 完成任务前应自检：

```txt
1. 当前任务是否完成？
2. 是否超出用户要求？
3. 是否违反阶段边界？
4. 是否新增依赖？
5. 是否执行 npm run build？
6. 是否更新 README / WORKLOG？
7. 是否有已知问题？
8. 是否需要用户确认下一步？
9. 是否给出 git commit 建议？
```

Agent 回复必须包含：

```txt
本轮完成内容
修改文件列表
新增 / 删除依赖
运行与构建结果
README / WORKLOG 是否更新
已知问题
下一步建议
Git 提交建议
```

---

## 31. 阶段验收标准

一个开发阶段可结束，至少满足：

```txt
1. 阶段目标全部完成
2. npm run build 通过
3. 新增页面可访问
4. 核心样式正常
5. 内容 schema 正常
6. 没有明显破坏其他页面
7. README 已更新
8. WORKLOG 已更新
9. 已知问题已记录
10. 用户可进行浏览器验收
```

阶段可提交后，再给出 Git 指令。

## 33. Bug 修复验收

修 bug 后必须确认：

```txt
1. bug 是否复现
2. 修复点在哪里
3. 是否影响其他模块
4. 是否新增依赖
5. 是否通过构建
6. 是否需要更新规范
7. 是否需要补充 WORKLOG
```

Agent 应输出：

```txt
问题原因
修复方式
修改文件
验证结果
剩余风险
```

```txt
3. 是否影响正文阅读
4. 是否影响暗色模式
5. 是否影响移动端

```

```txt

```

```txt
6. 是否记录到 WORKLOG
```

```txt
npm run build 失败

路由明显错误
新增依赖未说明
引入 Tailwind / shadcn/ui / CMS / 数据库

```

## 40. 最终原则

> Pixel Archive 的验收不是只看“有没有做出来”，而是看它是否能稳定运行、符合规范、保持风格、方便后续继续开发。  
> 每一轮开发结束时，都应该留下一个更清楚、更稳定、更容易继续维护的项目状态。
