## 2. 项目技术定位

Pixel Archive 是一个静态个人网站项目。

当前阶段技术定位：

```txt
静态站
内容驱动
Markdown / MDX 管理内容
Astro 负责页面生成
无后端
无数据库
无 CMS
本地先跑通
后续部署到 Cloudflare Pages
```

项目第一阶段优先目标：

```txt
结构稳定
依赖克制
构建可通过
视觉基础可用
Content Collections 可用
页面路由可用
```

---

## 3. 必须使用的技术栈

初版必须使用：

```txt
Astro
TypeScript
MDX
Astro Content Collections
原生 CSS
CSS Variables
```

用途说明：

| 技术 | 用途 |
|---|---|
| Astro | 静态站框架、页面路由、构建 |
| TypeScript | 类型约束、配置与内容读取辅助 |
| MDX | 支持 Markdown 中嵌入组件 |
| Astro Content Collections | 管理 Projects / Notes / Archive / Lab 内容 |
| 原生 CSS | 自定义视觉系统，不依赖大型 CSS 框架 |
| CSS Variables | 统一主题、颜色、字体、阴影、间距等设计变量 |

---

## 4. 当前阶段不使用的技术

未经用户明确确认，不允许引入：

```txt
Tailwind CSS
shadcn/ui
React UI 组件库
Vue UI 组件库
完整第三方 UI 框架
CMS
数据库
后端服务
复杂动画库
大型图标库
状态管理库
认证 / 登录系统
评论系统
搜索服务端
```

原因：

```txt
第一阶段目标是搭建轻量、稳定、可维护的静态个人网站。
过早引入大型依赖会增加维护成本，破坏当前“原生 CSS + 自研组件”的设计路线。
```

## 6. 依赖管理规则

## 6.1 新增依赖前必须说明

任何新增依赖前，Agent / 开发者必须说明：

```txt
1. 依赖名称
2. 用途
3. 解决的问题
4. 为什么不能用原生实现
5. 是否影响构建体积
6. 是否影响长期维护
7. 是否存在更轻量替代方案
8. 是否属于当前阶段范围
```

如果无法说明清楚，不应新增依赖。

---

## 6.2 第一阶段允许的核心依赖

第一阶段原则上只需要：

```txt
astro
@astrojs/mdx
typescript 相关依赖
```

如果 Astro 初始化模板自动生成必要依赖，可以保留。

---

## 6.3 第一阶段不建议新增的依赖

第一阶段不建议主动新增：

```txt
图标库
动画库
CSS 框架
UI 框架
搜索库
评论库
日期处理库
Markdown 高亮增强库
图片处理库
```

这些可在后续阶段按需评估。

---

## 6.4 依赖新增记录

如果确实新增依赖，必须在开发汇报中说明：

```txt
新增依赖
新增原因
安装命令
影响范围
是否更新 package.json / lock 文件
```

并更新 README 或 WORKLOG。

## 7. 开发阶段流程

Pixel Archive 开发必须小步推进。

推荐阶段：

```txt
Phase 0：项目初始化
Phase 1：基础架构与视觉系统
Phase 2：首页初版
Phase 3：页面空壳与路由
Phase 4：Content Collections 与占位内容
Phase 5：Projects 列表与详情
Phase 6：Notes 列表与详情
Phase 7：Archive / Lab / About / 404 Polish
Phase 8：部署准备
```

每个阶段应满足：

```txt
阶段目标清晰
修改范围可控
本地可运行
构建可验证
README / WORKLOG 更新
能说明下一步
```

## 9. 目录与文件创建规则

项目目录结构遵循：

```txt
05_模块化架构规范.md
```

内容文件、图片文件、slug、frontmatter 遵循：

```txt
04_内容与文件管理规范.md
```

本文件只规定开发执行层面的要求：

```txt
不要随意创建临时目录
不要使用 test / demo / aaa 等随意命名
不要创建重复功能目录
不要把业务组件、基础组件、页面文件混放
不要把图片随意放在项目根目录
```

如果确实需要新增目录，必须说明：

```txt
新增目录路径
所属模块
用途
为什么现有目录不适合
```

## 10. 样式开发执行规则

样式文件职责：

```txt
theme.css：主题变量、颜色、字体、阴影、圆角等 tokens
global.css：reset、body、基础布局、链接、滚动条等全局样式
markdown.css：Markdown 正文排版
组件内 style：组件局部样式
```

执行要求：

```txt
优先使用 CSS Variables
不要在组件中大量硬编码颜色
不要滥用 !important
不要让 markdown.css 影响首页或项目卡片
不要把大量业务组件样式写进 global.css
不要为小问题引入 CSS 框架
```

具体视觉标准见：

```txt
03_视觉风格规范.md
```

```txt

```

## 12. 开发前检查

Agent / 开发者每次开始前，应先确认：

```txt
1. 当前任务属于哪个阶段？
2. 当前任务是否超出阶段范围？
3. 当前任务需要读取哪些规范？
4. 是否会新增依赖？
5. 是否会改变技术栈？
6. 是否会改变目录结构？
7. 是否会改变路由？
8. 是否会改变 Content Collections schema？
9. 是否会改变组件接口？
10. 是否需要先询问用户？
```

如果答案涉及重大变化，应先说明影响，不要直接实现。

## 13. 变更控制规则

允许直接处理的小改动：

```txt
拼写错误
路径小错误
类型定义小错误
样式细节小 bug
构建报错修复
README / WORKLOG 小更新
```

不允许未经确认直接改变：

```txt
技术栈
目录结构
页面结构
内容模型
视觉主方向
部署平台
是否引入 Tailwind / React / CMS
是否新增后端能力
是否改变路由命名
是否改变 schema 字段
```

如果发现原规划不合理，应输出：

```txt
发现的问题
为什么原规划不适合
建议修改方案
影响范围
是否需要用户确认
```

## 14. 构建与运行命令

第一阶段至少应支持：

```bash
npm install
npm run dev
npm run build
```

如果项目配置了检查命令，可使用：

```bash
npm run check
npm run lint
```

Agent 开发完成后，应尽量执行：

```bash
npm run build
```

构建失败不能声称完成。

如果构建失败，必须说明：

```txt
失败命令
错误摘要
可能原因
已尝试修复
剩余问题
```

## 15. README 规范

每轮开发后，应更新 README。

README 至少包含：

```txt
项目简介
技术栈
运行方式
目录结构
当前完成内容
下一步计划
```

README 不应记录过长开发细节。详细开发过程应写入 WORKLOG。

---

## 16. WORKLOG 规范

建议建立：

```txt
WORKLOG.md
```

每轮开发后记录：

```txt
日期
阶段
修改文件
完成内容
遇到的问题
构建结果
下一步建议
```

WORKLOG 作用：

```txt
保留开发过程
方便后续 Agent 接续
避免 README 过长
记录决策变化
记录已知问题
```

如果用户后续明确不需要 WORKLOG，可取消；默认建议保留。

---

## 17. Agent 每轮输出格式

Agent 完成开发后，必须输出：

```txt
1. 本轮完成内容
2. 修改文件列表
3. 新增 / 删除依赖
4. 运行与构建结果
5. README / WORKLOG 是否更新
6. 已知问题
7. 下一步建议
8. Git 提交建议
```

不要只输出：

```txt
已完成
```

## 18. Git 规则

## 18.1 Commit 时机

建议每个阶段验收通过后再 commit。

不建议：

```txt
每改一个小文件就 commit
构建失败时 commit
未更新 README / WORKLOG 就 commit
```

---

## 18.2 Commit message

建议使用中文 commit message。

示例：

```txt
初始化 Pixel Archive 项目骨架
实现基础视觉系统与布局
实现首页档案终端初版
实现 Content Collections 与占位内容
实现项目列表与详情页
实现 Notes 列表与文章页
完善 Archive 与 Lab 页面
补充部署配置与 README
```

---

## 18.3 Push 规则

默认规则：

```txt
git push 由用户手动执行
Agent 只提供 commit / push 指令
```

建议输出：

```bash
git status
git add .
git commit -m "初始化 Pixel Archive 项目骨架"
git push origin main
```

除非用户明确要求，否则 Agent 不应自动 push。

## 20. 性能与维护原则

初版应保持轻量。

要求：

```txt
不加载大图作为首页核心
不依赖复杂动画
不引入大型 UI 库
不引入多余字体文件
不引入不必要 JS
Markdown 页面优先静态生成
内容结构长期稳定
```

后续如需加入：

```txt
搜索
代码高亮增强
RSS
SEO 增强
图片优化
```

应分阶段处理。

---

## 21. 可访问性基础要求

第一阶段至少保证：

```txt
链接可聚焦
按钮有可见 hover / focus 状态
颜色对比基本可读
图片有 alt
主题切换按钮有 aria-label
导航结构语义清晰
页面 title 正确
```

不要为了像素风牺牲基本可访问性。

---

## 22. 移动端基础要求

项目以桌面体验优先，但移动端不能崩。

第一阶段至少保证：

```txt
导航在窄屏可换行或折叠
首页终端不横向溢出
项目卡片可单列显示
Notes 正文宽度适配
代码块可横向滚动
表格可横向滚动
```

更详细验收见：

```txt
09_验收与测试规范.md
```

---

## 23. 不应做的事情清单

开发过程中不要做：

```txt
把网站做成普通博客模板
把网站做成全站 NES UI
把网站做成 Windows 98 仿站
把网站做成纯终端黑绿风
使用 Tailwind 重写设计系统
引入 shadcn/ui
引入大型组件库
把首页写成个人简历
在 Notes 正文使用像素字体
让视觉效果影响长文阅读
把图片随意放在根目录
用中文文件名创建内容
用 test/demo/aaa 命名占位文件
构建失败仍声称完成
```

## 25. 最终原则

> Pixel Archive 的技术与开发执行应当依赖克制、阶段清晰、变更可控、构建可验收。  
> 每轮开发都必须能说明改了什么、为什么改、如何运行、如何验证，以及下一步应该做什么。
