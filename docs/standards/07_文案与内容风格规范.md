# 07_文案与内容风格规范

版本：v1.0  
项目：Pixel Archive  
位置：`docs/standards/07_文案与内容风格规范.md`  
用途：规范 Pixel Archive 的整体文案气质、语言使用、首页 UI 短句、项目简介、Notes 摘要、Archive 存档点、Lab 文案、About 文案、空状态、404 文案和 Agent 写作边界。  

---

## 1. 本文档的作用

本文档负责回答：

```txt
1. Pixel Archive 的文字应该是什么气质？
2. 中文和英文如何混用？
3. 首页 UI 文案应该怎么写？
4. 项目简介应该怎么写？
5. Notes 摘要应该怎么写？
6. Archive 存档点应该怎么写？
7. Lab 实验内容应该怎么写？
8. About 页面应该写到什么程度？
9. 空状态和 404 文案应该怎么写？
10. Agent 写文案时应该避免什么？
```

本文档不负责定义：

```txt
技术栈
视觉色彩
文件命名
模块架构
组件 props
部署流程
验收命令
```

这些由其他专项规范负责。

---

## 2. 文案一句话定义

Pixel Archive 的文案应该是：

> 简短、克制、有一点像素档案感，但不装腔、不鸡汤、不简历化。

它应该像：

```txt
一个安静的个人档案终端
一个长期维护的项目存档
一个有轻微游戏隐喻的知识空间
```

它不应该像：

```txt
求职简历
企业官网
营销落地页
自我介绍长文
二次元角色设定页
纯游戏菜单
中二故事设定
```

---

## 3. 总体语气原则

Pixel Archive 的文字应遵循：

```txt
短
准
安静
克制
有个人风格
有档案感
不过度解释
不主动自夸
```

避免：

```txt
宏大叙事
过度热血
过度谦虚
过度可爱
过度正式
过度营销
过度技术黑话
```

推荐感受：

```txt
像一个私人工作台
像一个项目存档库
像一个安静的 save file
像一个正在慢慢生长的网站
```

---

## 4. 中文与英文使用规则

## 4.1 主语言

网站初版以中文为主。

适合中文的区域：

```txt
Notes 正文
项目详情说明
Archive 说明
About 简短介绍
长段落
技术解释
开发日志
```

---

## 4.2 英文使用

英文主要用于：

```txt
导航
页面标题
项目名
UI 短句
状态词
标签
技术名词
像素终端提示
```

示例：

```txt
Home
Projects
Notes
Archive
Lab
About

Pixel Archive
SAVE SLOT 01
archive online
save loaded
new slot unlocked
mode quiet
```

---

## 4.3 中英文混用原则

允许中英文混用，但要自然。

推荐：

```txt
Pixel Archive 是一个用于存放项目、笔记和实验内容的个人档案馆。
当前阶段主要在搭建 Astro + MDX 的静态站框架。
```

不推荐：

```txt
这个 Website 是一个用于 Showcase 自己 Personal Works 的 Portfolio System。
```

技术名词保持英文即可，不必强行翻译。

---

## 5. 首页文案风格

首页不是自我介绍，而是入口场景。

首页文案应像 UI 提示，不像文章段落。

---

## 5.1 首页允许的文案类型

```txt
站点名
终端状态
菜单入口
存档槽标题
短状态条
少量说明
```

示例：

```txt
Pixel Archive

archive online
save loaded
new slot unlocked
mode quiet
building personal website

> Projects
> Notes
> Archive
> Lab
> About

SAVE SLOT 01
Project Slot
WIP

SAVE SLOT 02
Research Slot
locked

SAVE SLOT 03
Lab Slot
empty
```

---

## 5.2 首页不应出现

```txt
大家好，我是……
欢迎来到我的个人网站……
这里是我的个人主页……
我热爱技术、热爱生活……
本人具备扎实的……
这是一个展示我个人能力的平台……
```

首页不写传统个人介绍。

---

## 5.3 首页短句风格

推荐：

```txt
archive online
loading index
save loaded
new file created
slot waiting
mode quiet
building
reading
drafting
system idle
```

中文短句也可以：

```txt
档案已载入
新的存档槽等待写入
这里还在搭建
暂时安静运行中
```

不推荐：

```txt
欢迎来到我的梦幻像素世界
这是我的精神家园
开启属于你的冒险吧
探索我的人生旅程
```

---

## 6. 导航文案

导航保持英文，短而稳定。

固定导航：

```txt
Home
Projects
Notes
Archive
Lab
About
```

不建议替换为：

```txt
主页
项目展示
我的文章
时间轴
实验室
关于我
```

也不建议过度游戏化为：

```txt
Start
Inventory
Quest Log
Save Room
Secret Lab
Player Info
```

可以在首页菜单中轻微使用终端风：

```txt
> Projects
> Notes
> Archive
> Lab
> About
```

但顶部导航保持清晰。

---

## 7. Project 文案规范

Projects 文案目标：

```txt
说明项目是什么
解决什么问题
当前状态是什么
使用什么技术
做到什么程度
后续可能做什么
```

不需要包装成营销文案。

---

## 7.1 Project description

Project description 应该短，建议 1–2 句。

推荐结构：

```txt
这是一个……工具 / 项目 / 原型，用于……。
当前实现了……，后续计划……。
```

示例：

```txt
ScholarLens AI 是一个面向英文论文阅读的 PDF 辅助工具，支持原文阅读、Markdown 内容层、翻译、笔记和 AI 对话。当前重点是解析源切换、VLM Parser 和阅读体验优化。
```

更短示例：

```txt
一个用于整理项目、笔记和实验内容的像素风个人档案网站。
```

---

## 7.2 Project 不应写成

```txt
本项目具有重要意义……
本系统创新性地……
本人通过该项目充分展示了……
这是一个非常优秀的……
该项目极大提升了……
```

除非是正式论文或项目报告，否则项目页不要写论文摘要腔。

---

## 7.3 Project status 文案

状态建议：

```txt
WIP
Done
Prototype
Paused
```

对应显示可以保留英文，不强制翻译。

如果需要中文说明：

| Status | 中文说明 |
|---|---|
| WIP | 正在开发 |
| Done | 阶段完成 |
| Prototype | 原型 |
| Paused | 暂停 |

不要使用太多状态：

```txt
Almost Done
Doing
Pending
Finished
Experimental
Archived
```

除非后续统一扩展。

---

## 7.4 Project Detail 正文结构建议

项目详情页文案可按以下结构：

```txt
Overview
Why
Current Status
Tech Stack
Notes
Next
```

中文写法：

```txt
## Overview

简短说明项目是什么。

## Why

说明为什么做这个项目。

## Current Status

说明当前已经做到哪里。

## Tech Stack

列出技术栈。

## Notes

记录关键设计和问题。

## Next

说明后续计划。
```

不需要每个项目都写很长。

---

## 8. Notes 文案规范

Notes 是主要内容沉淀区，文字应比首页更自然。

Notes 分为：

```txt
Article
Log
Note
```

不同类型文风不同。

---

## 8.1 Article

Article 是正式文章。

文风：

```txt
结构完整
逻辑清楚
语气稳定
可以较正式
避免口水化
```

适合：

```txt
技术总结
论文阅读
系统设计分析
工具使用指南
长期可读内容
```

Article 摘要建议：

```txt
本文整理……，重点包括……，适合作为……的参考。
```

不要写：

```txt
今天随便记一下……
感觉挺有意思……
以后再说……
```

---

## 8.2 Log

Log 是阶段记录。

文风：

```txt
具体
直接
记录发生了什么
记录为什么这样做
记录下一步
```

适合：

```txt
项目开发日志
阶段总结
踩坑记录
版本变化
```

Log 摘要建议：

```txt
记录 Pixel Archive 第一阶段规划过程，包括页面结构、视觉方向和内容系统的初步确定。
```

---

## 8.3 Note

Note 是学习笔记或临时整理。

文风：

```txt
轻量
清楚
可以不完整
保留上下文
方便以后继续整理
```

适合：

```txt
概念解释
学习片段
工具命令
阅读摘录
临时方案
```

Note 摘要建议：

```txt
整理 Astro Content Collections 的基本用法，以及它在 Pixel Archive 中的内容组织方式。
```

---

## 9. Archive 文案规范

Archive 是存档点，不是日记。

Archive 文案应像：

```txt
某个阶段的保存记录
某个项目的里程碑
某次方向确定
某个版本完成
```

语气可以比 Notes 更短、更像系统记录。

---

## 9.1 Archive title

推荐：

```txt
Started Pixel Archive
Pixel Archive Planning Completed
ScholarLens AI VLM Parser Milestone
Graduation Project Wrapped
```

也可以使用中文：

```txt
Pixel Archive 启动
ScholarLens AI VLM Parser 阶段完成
毕业设计阶段收尾
```

---

## 9.2 Archive description

建议 1 句即可。

推荐：

```txt
完成 Pixel Archive 的项目定位、页面结构、视觉方向和文档体系规划。
```

```txt
完成 ScholarLens AI VLM Parser 的真实接口调用和分批解析验证。
```

不推荐：

```txt
今天我终于开始了一个很有意义的新旅程……
这一天对我来说意义非凡……
```

---

## 9.3 Archive 类型文案

Archive type：

```txt
save-point
milestone
project-log
```

显示文案可以是：

```txt
SAVE POINT
MILESTONE
PROJECT LOG
```

或中文：

```txt
存档点
里程碑
项目记录
```

不要每条 Archive 写成长篇文章。详细内容应写到 Notes。

---

## 10. Lab 文案规范

Lab 是实验区，可以比其他页面更自由。

Lab 文案允许：

```txt
轻松一点
短一点
有实验感
有占位感
有一些游戏 / 像素语气
```

但不要变成无意义玩梗。

---

## 10.1 Lab description

推荐：

```txt
A small UI experiment for pixel-style cards and terminal-like status labels.
```

```txt
一个用于测试像素按钮、硬边框和低饱和紫绿色组合的小实验。
```

不推荐：

```txt
这里是我的神秘实验室，里面充满未知和惊喜……
```

---

## 10.2 Lab status

Lab status：

```txt
idea
prototype
done
```

显示文案：

```txt
IDEA
PROTOTYPE
DONE
```

或中文：

```txt
想法
原型
完成
```

---

## 11. About 文案规范

About 页面应极短、克制，不做正式简历。

目标：

```txt
告诉访客这个网站是什么
提供少量个人方向
提供联系入口
```

不写：

```txt
完整履历
技能清单堆叠
求职自我评价
长篇个人故事
过度情绪化描述
```

---

## 11.1 About 推荐结构

```txt
Pixel Archive is a small personal archive for projects, notes, save points and visual experiments.

中文补充一句：
这里主要存放项目、学习记录、工具原型和一些视觉实验。

Links:
GitHub / Email / Other
```

如果使用中文：

```txt
Pixel Archive 是一个个人档案馆，用来存放项目、笔记、阶段存档和一些实验性内容。这里不会写太长的自我介绍，只保留一些正在做的事情和可以继续追踪的线索。
```

---

## 11.2 About 不推荐文案

```txt
我是一名热爱技术、积极进取、具有良好团队协作能力的人……
本人熟练掌握……
未来希望在……
欢迎各位老师同学批评指正……
```

这些更像简历或论文致谢，不适合 Pixel Archive。

---

## 12. 空状态文案规范

空状态不能像系统报错，也不要太冷冰冰。

推荐短句：

```txt
No files loaded yet.
This drawer is still empty.
New archive slot waiting.
Nothing here, for now.
Waiting for the next save point.
No notes archived yet.
```

中文短句：

```txt
这里还没有内容。
新的存档槽等待写入。
暂时没有载入文件。
下一条记录还在路上。
```

避免：

```txt
暂无数据
数据为空
列表为空
加载失败
无内容
```

除非确实是系统错误。

---

## 13. 404 文案规范

404 页面应符合 Pixel Archive 气质。

推荐主标题：

```txt
save file not found
lost archive
archive entry missing
```

推荐说明：

```txt
The file you are looking for is not in this archive.
```

```txt
This path points to an empty save slot.
```

中文说明：

```txt
你访问的路径没有对应的存档。
```

按钮：

```txt
Return Home
Back to Archive
Load Home Slot
```

不推荐：

```txt
哎呀，页面不见了！
Oops! Something went wrong!
你迷路啦！
```

不是不能用轻松语气，但要避免普通模板感。

---

## 14. 按钮与链接文案

按钮文案应短。

推荐：

```txt
Open
Read
View Project
Read Note
Back Home
View on GitHub
Open Demo
Load Slot
```

不推荐：

```txt
点击这里查看更多详细信息
马上开始探索我的项目
进入这个神奇的页面
```

链接文案尽量说明目的，不要只写：

```txt
这里
点击
更多
```

推荐：

```txt
Read the note
View project
Open GitHub
```

---

## 15. Footer 文案规范

Footer 可以包含：

```txt
Pixel Archive
Built with Astro
年份
GitHub
一句短句
```

短句示例：

```txt
archive online
quietly building
save often
files remain
```

中文短句：

```txt
安静存档中
慢慢搭建中
记得保存
```

不要写很长。

---

## 16. Status / Badge 文案规范

状态文案应短、稳定、可复用。

Project status：

```txt
WIP
Done
Prototype
Paused
```

Note type：

```txt
Article
Log
Note
```

Archive type：

```txt
SAVE POINT
MILESTONE
PROJECT LOG
```

Lab type：

```txt
PIXEL
UI
MINECRAFT
IMAGE
DEMO
OTHER
```

避免同一状态出现多种写法：

```txt
WIP
In Progress
Doing
Developing
```

应统一。

## 17. 页面标题规范

页面主标题保持英文：

```txt
Projects
Notes
Archive
Lab
About
```

页面副标题可以中英文混合。

示例：

```txt
Projects
一些正在完成、已经完成和暂时搁置的项目。

Notes
文章、学习记录和开发日志。

Archive
重要节点和阶段存档。

Lab
一些视觉、像素和小工具实验。
```

---

## 18. 分类与标签文案

Category 可以使用英文短语：

```txt
AI / ML
Aerospace / Control
Tools / Web
Creative / Lab
Web
Paper Reading
Development Log
Personal
```

Tag 使用小写英文：

```txt
astro
markdown
typescript
pdf
vlm
flight-control
system-identification
pixel-ui
```

不要在 tag 中使用中文、空格或随意大小写。

具体命名由：

```txt
04_内容与文件管理规范.md
```

定义。

---

## 19. 文案长度控制

不同区域建议长度：

| 区域 | 建议长度 |
|---|---|
| 首页状态短句 | 1–5 个词 |
| 菜单说明 | 1 短句 |
| Project description | 1–2 句 |
| Note description | 1 句 |
| Archive description | 1 句 |
| Lab description | 1–2 句 |
| About intro | 2–4 句 |
| EmptyState title | 1 短句 |
| EmptyState description | 1 句 |
| 404 description | 1 句 |

如果某段文案超过 5 行，应考虑移动到详情页或 Notes。

---

## 20. 禁用或慎用表达

## 20.1 禁止简历腔

避免：

```txt
本人熟练掌握……
具备扎实的……
拥有良好的……
具有较强的……
在校期间积极参与……
```

---

## 20.2 禁止营销腔

避免：

```txt
极大提升……
高效赋能……
一站式解决……
全方位打造……
革命性体验……
```

---

## 20.3 禁止过度中二

避免：

```txt
这里封印着我的灵魂碎片……
开启命运之门……
踏上未知旅途……
```

---

## 20.4 禁止模板废话

避免：

```txt
欢迎来到我的个人网站
这里会记录我的点点滴滴
感谢你的访问
更多内容敬请期待
```

可以表达“内容后续补充”，但要更像 Pixel Archive：

```txt
New archive slot waiting.
```

## 23. Agent 写作规则

Agent 在生成文案时必须遵守：

```txt
先判断文案属于哪个页面 / 组件
优先使用短句
不要写传统个人简介
不要写简历腔
不要写营销腔
不要主动扩展人设
不要编造用户真实经历
不要编造项目成果
不要把占位内容写得像真实已完成内容
```

如果缺少真实信息，应写成占位但不欺骗：

```txt
Placeholder project for testing the Pixel Archive content system.
```

或中文：

```txt
用于测试 Pixel Archive 内容系统的占位项目。
```

---

## 24. 占位内容文案规则

占位内容必须明确是占位，不能假装真实。

推荐：

```txt
This is a placeholder item for the first Pixel Archive build.
```

```txt
用于验证 Projects 页面结构和内容字段的占位项目。
```

不推荐：

```txt
这是一个已经完整实现的优秀项目……
```

如果是用户真实项目，但内容暂时未填，可以写：

```txt
A real project entry. Details will be added later.
```

或：

```txt
真实项目条目，详细内容后续补充。
```

## 25. 文案修改规则

后续修改文案时，应考虑：

```txt
是否仍然符合 Pixel Archive 气质
是否过长
是否过度正式
是否过度游戏化
是否影响阅读
是否与页面功能匹配
是否和已有状态词冲突
```

如果修改涉及全站固定词汇，例如：

```txt
Projects / Notes / Archive / Lab
WIP / Done / Prototype / Paused
Article / Log / Note
```

必须同步检查：

```txt
04_内容与文件管理规范.md
06_组件设计规范.md
```

## 27. 文案验收标准

检查文案时，至少确认：

```txt
1. 首页没有写成个人简介
2. 首页文案短，像 UI 提示
3. Projects 文案说明项目，不营销
4. Notes 摘要清楚，不口水化
5. Archive 像存档点，不像日记
6. Lab 可以轻松，但不乱玩梗
7. About 极短克制，不像简历
8. 404 有 lost archive / save file not found 气质
9. 空状态不使用生硬“暂无数据”
10. 状态词保持统一
11. 中英文混用自然
12. 没有编造用户经历和项目成果
13. 没有过度中二或模板腔
14. 文案与视觉风格一致
15. 文案长度符合页面位置
```

---

## 28. 最终原则

> Pixel Archive 的文案应像一个安静运行的私人档案终端：短、清楚、克制、有一点存档感。  
> 它不需要解释太多，也不需要证明太多；只需要把项目、笔记和存档点清楚地放在合适的位置。

## 29. Phase 22 语言角色收口

全站中英混排按“系统层 / 内容层”分工：

```txt
英文系统层：导航、索引、状态、badge、技术标签、坐标、操作入口
中文内容层：页面副标题、标题、摘要、项目说明、Archive 描述、正文
```

稳定操作文案使用：

```txt
LOAD PROJECT →
OPEN NOTE →
OPEN RELATED FILE →
OPEN EXPERIMENT →
OPEN RECORD →
```

规则：

```txt
同类操作不混用 OPEN / 查看 / 详情
同类状态不混用 WIP / 进行中
技术标签继续使用小写英文
英文 UI 标签保持短、稳定，可使用大写
中文正文和长标题不为了像素感改成英文或强制大写
```
