# 06_组件设计规范

版本：v1.1
项目：Pixel Archive  
位置：`docs/standards/06_组件设计规范.md`  
用途：规范 Pixel Archive 的组件分层、组件命名、组件职责、props 设计、视觉变体、UI 组件与业务组件边界，确保组件系统清晰、可复用、可维护。  

---

## 1. 本文档的作用

本文档负责回答：

```txt
1. Pixel Archive 需要哪些组件？
2. 哪些是基础 UI 组件，哪些是业务组件？
3. 组件应该如何命名？
4. 组件 props 应该怎么设计？
5. 组件可以有哪些 variant / size / status？
6. 组件之间如何组合？
7. 哪些组件不能包含业务逻辑？
8. 如何避免组件重复、混乱、耦合？
```

本文档不负责定义：

```txt
技术栈和依赖管理
整体视觉色彩系统
内容文件命名
模块依赖方向
文案具体写法
部署流程
验收命令
```

这些分别由其他专项规范负责。

---

## 2. 组件设计总原则

Pixel Archive 的组件系统应遵循：

```txt
小而清晰
职责单一
命名准确
props 明确
视觉统一
业务边界清楚
不为抽象而抽象
不为复用而过度复杂
```

核心原则：

> UI 组件提供通用视觉，业务组件表达具体内容，页面负责组织数据，组件不应擅自读取全局内容。

---

## 3. 组件分层

Pixel Archive 组件分为三层：

```txt
基础 UI 组件
业务展示组件
页面组合组件
```

对应关系：

| 层级 | 路径 | 作用 |
|---|---|---|
| 基础 UI 组件 | `src/components/ui/` | 通用按钮、卡片、标签、徽章、空状态等 |
| 业务展示组件 | `src/components/home/`、`project/`、`notes/`、`archive/`、`lab/` | 展示特定业务内容 |
| 页面组合组件 | 少量位于业务目录中 | 将多个业务组件组合成页面区域 |

## 5. 组件命名规则

组件使用 PascalCase：

```txt
ProjectCard.astro
NoteCard.astro
ArchiveTerminal.astro
ThemeToggle.astro
```

推荐命名：

```txt
职责明确
名词优先
能看出所属业务
能看出组件用途
```

推荐：

```txt
ProjectCard
ProjectGrid
NoteList
SaveSlot
ArchiveTimeline
LabCard
EmptyState
StatusPill
```

不推荐：

```txt
Box
Block
Panel
Panel2
ContainerFinal
NewCard
CommonCard
Thing
MyComponent
DemoComponent
TestCard
```

如果一个组件名称无法准确说明职责，说明组件可能需要拆分或重新设计。

## 6. Props 设计原则

组件 props 应该：

```txt
少而明确
类型稳定
命名直观
避免传入整个庞大对象
避免让组件自己猜字段
避免隐式依赖外部全局数据
```

推荐：

```txt
<ProjectCard
  title={project.data.title}
  description={project.data.description}
  status={project.data.status}
  tags={project.data.tags}
  href={`/projects/${project.slug}/`}
/>
```

不推荐：

```txt
<ProjectCard project={rawProject} />
```

如果传入整个对象，应确保该对象类型稳定且属于当前模块内部约定。

---

## 7. Props 命名规范

通用命名：

| prop | 用途 |
|---|---|
| `title` | 标题 |
| `description` | 简短描述 |
| `href` | 链接地址 |
| `label` | 短文本标签 |
| `variant` | 视觉变体 |
| `size` | 尺寸 |
| `status` | 状态 |
| `tags` | 标签数组 |
| `date` | 日期 |
| `active` | 是否激活 |
| `disabled` | 是否禁用 |
| `featured` | 是否重点展示 |
| `className` | 可选额外类名，谨慎使用 |

避免使用：

```txt
text1
text2
data
item
thing
obj
info
config
custom
```

除非有明确类型和上下文。

---

## 8. Variant 设计原则

组件可以通过 `variant` 控制视觉变体，但不要无限扩展。

推荐通用 variant：

```txt
default
soft
pixel
outline
ghost
terminal
```

状态类 variant：

```txt
success
warning
danger
muted
```

不推荐：

```txt
purpleBigCardWithShadow
homeSpecialButtonFinal
projectCard2
greenStyle
styleA
styleB
```

如果 variant 名称越来越具体，说明应该拆分组件或重新设计。

---

## 9. Size 设计原则

组件 size 保持少量枚举。

推荐：

```txt
sm
md
lg
```

不推荐：

```txt
tiny
mini
small
normal
medium
large
huge
xl
xxl
```

第一阶段大多数组件只需要 `sm / md / lg`，甚至不需要 size。

## 10. 基础 UI 组件规范

基础 UI 组件位于：

```txt
src/components/ui/
```

UI 组件必须保持无业务含义。

它们可以知道：

```txt
label
href
variant
size
disabled
active
```

它们不应该知道：

```txt
Project
Note
Archive
Lab
Content Collections
页面路由业务
项目状态含义
文章类型含义
```

## 11. Button

路径：

```txt
src/components/ui/Button.astro
```

职责：

```txt
通用按钮或链接按钮
支持普通导航和页面操作
```

建议 props：

```ts
type ButtonProps = {
  href?: string;
  label?: string;
  variant?: "default" | "pixel" | "outline" | "ghost" | "terminal";
  size?: "sm" | "md" | "lg";
  disabled?: boolean;
  external?: boolean;
};
```

视觉要求：

```txt
默认克制
pixel 变体可使用像素边框
hover 有轻微反馈
focus 状态可见
不要过度动画
```

禁止：

```txt
Button 内部写死 Projects / Notes 路由
Button 内部写死业务文案
Button 内部读取导航配置
```

禁止：

```txt

```

## 26. 组件样式规则

组件样式原则：

```txt
组件局部样式写在组件内
颜色使用 CSS Variables
不要硬编码大量色值
不要滥用 !important
不要依赖其他组件内部 DOM
不要让一个组件的 CSS 影响全站
```

推荐：

```css
.card {
  background: var(--color-surface);
  border: 1px solid var(--color-border);
}
```

不推荐：

```css
.card {
  background: #fffaf0;
  color: #2d2a26;
}

.project-card .markdown h2 {
  ...
}
```

全局视觉变量由：

```txt
03_视觉风格规范.md
```

定义。

## 26.1 稳定颜色映射

Phase 10 验收后的组件色彩职责：

| 组件 / 元素 | 颜色角色 |
|---|---|
| Header Logo | `--color-heading` + `--color-primary` 像素方块 |
| Header Active | `--color-accent` 文字与断续下划线 |
| ArchiveTerminal | paper surface、`--color-heading` 标题、green prompt、red arrow / pixel |
| StatusBar | raised surface；green / violet / warm 三类状态 |
| SaveSlot | building=primary、locked=warm、empty=violet；动作统一 accent |
| ProjectCard | primary 角标与 hover pixel shadow、warm WIP、accent action |
| NoteCard | info / 黛蓝标题、primary tags、accent 仅作 hover |
| ArchiveEntry | warm 年份与节点、primary 月份、accent 相关文件动作 |
| LabCard | violet badge / 像素块 / hover shadow、accent 只作关键动作 |
| About | cyan / primary 小标签，主体保持中性 |

规则：

```txt
普通边框必须使用 border 语义变量，不使用紫色。
标题统一使用 heading token。
小字号彩色文字使用 primary-text / accent-text / warm-text 对比度 token。
彩色像素阴影只出现在 hover、featured 或 active。
组件状态不能只依赖颜色，必须保留文字或像素信号。
页面栏目差异通过 data-section 变量实现，不逐页硬编码 HEX。
```

## 28. 组件拆分规则

当组件出现以下情况，应考虑拆分：

```txt
组件超过 200–300 行
props 数量过多
同时处理多个业务区域
样式难以理解
HTML 结构层级过深
某部分可以被多个页面复用
组件名称无法准确描述职责
```

拆分优先级：

```txt
先拆业务区域
再拆通用 UI
再拆样式复杂部分
不要过度拆空壳
```

## 36. PixelCover 组件族

Phase 19 在 `src/components/ui/` 增加：

```txt
PixelCover.astro
PixelMatrix.astro
PixelBars.astro
PixelStatusMark.astro
```

`PixelCover` 只接收展示数据：

```ts
type PixelCoverProps = {
  variant:
    | "file-matrix"
    | "terminal-strip"
    | "save-chip"
    | "timeline-fragment"
    | "experiment-grid";
  section: "projects" | "notes" | "archive" | "lab" | "home";
  status?: string;
  tags?: string[];
  seed: string;
  compact?: boolean;
  label?: string;
};
```

规则：

```txt
业务组件负责选择 variant 并传入 metadata。
PixelCover 不读取 Content Collections 或路由状态。
相同 seed 在构建和刷新后必须生成相同结果。
所有变体使用 CSS Variables 和栏目 token，不硬编码 HEX。
内容提供真实 cover 时优先使用真实图片。
状态必须保留文字，不能只靠颜色表达。
```

当前组合关系：

```txt
ProjectCard / ProjectVisual -> Project type mapping -> PixelCover
LabCard -> Lab type mapping -> PixelCover
SaveSlot -> save-chip variant -> PixelCover
PixelCover -> PixelMatrix + PixelBars + PixelStatusMark
```

## 37. 侧边辅助组件

Phase 20 增加：

```txt
src/components/ui/SideRail.astro
src/components/notes/OnThisPage.astro
src/components/notes/ReadingProgress.astro
```

职责边界：

```txt
SideRail：显示页面提供的低对比 label/value，不读取页面或内容集合。
OnThisPage：接收 Astro render 输出的 heading 数组，输出锚点并标记当前章节。
ReadingProgress：接收目标 selector，使用原生 scroll/resize 事件更新进度。
```

规则：

```txt
桌面端可 sticky，移动端必须折叠、转静态或隐藏。
不得遮挡正文，不得把正文宽度压缩到不可读。
目录锚点必须指向真实 heading id。
阅读进度必须使用 progressbar 语义与 aria-valuenow。
滚动监听必须使用 requestAnimationFrame 节流。
不得新增动画、滚动或状态管理依赖。
```

## 38. UI 字体工具类

英文系统层组件统一使用：

```txt
.ui-brand：站点品牌字样
.ui-label：导航、状态、badge、计数、系统操作入口
```

组件仍负责字号、颜色和布局；工具类只统一 `--font-ui`、字距、大小写和数字字形。业务内容标题、中文描述和 Markdown 正文不得继承这两个工具类。
