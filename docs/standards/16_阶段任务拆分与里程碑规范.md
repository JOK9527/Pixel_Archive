# 16_阶段任务拆分与里程碑规范

版本：v1.0  
项目：Pixel Archive  
位置：`docs/standards/16_阶段任务拆分与里程碑规范.md`  
用途：将 Pixel Archive 的开发过程拆分为可执行、可验收、可提交的小阶段，明确每个阶段的目标、输入、输出、禁止事项、验收标准和建议 commit message。  

---

## 1. 本文档的作用

本文档负责回答：

```txt
1. Pixel Archive 应该按什么阶段开发？
2. 每个阶段具体做什么？
3. 每个阶段不做什么？
4. 每个阶段完成后应该产出什么？
5. 每个阶段如何验收？
6. 每个阶段完成后建议如何 commit？
```

本文档不是完整项目说明。  
项目定位见：

```txt
00_项目开发计划书.md
```

Agent 执行模板见：

```txt
10_Agent执行指令模板.md
```

## 2. 阶段开发总原则

Pixel Archive 采用小步阶段式开发。

原则：

```txt
先骨架
再视觉
再首页
再页面
再内容
再列表
再详情
再部署
再扩展
```

不允许：

```txt
第一阶段就做搜索
第一阶段就做评论
第一阶段就做 CMS
第一阶段就做复杂像素房间
第一阶段就接 GitHub API
第一阶段就做完整 SEO
```

每个阶段都要做到：

```txt
目标明确
范围有限
构建通过
记录完整
可回退
可提交
```

```txt

Phase 12：像素头像 / 首页视觉增强
Phase 13：Lab 内容扩展
```

## 17. 阶段进入条件

进入下一阶段前必须满足：

```txt
当前阶段目标完成
npm run build 通过
README / WORKLOG 更新
已知问题记录
用户验收或明确允许继续
```

如果构建失败，不进入下一阶段。

---

## 18. 阶段暂停规则

如果出现以下情况，应暂停阶段推进：

```txt
技术栈需要改变
依赖需要新增
内容模型需要重设
视觉方向需要大改
路由结构需要重设
构建无法通过
用户提出新的核心需求
```

暂停时应先输出影响分析。

---

---

## 20. 最终原则

> Pixel Archive 的开发应像存档点一样推进：每个阶段都能运行、能构建、能解释、能回退。  
> 阶段边界越清楚，Agent 越不容易跑偏，后续维护也越稳定。
