# Pixel Archive 部署与日常更新操作指南

> 当前生产地址：<https://pixel-archive.pixelarchive.workers.dev>  
> GitHub 仓库：<https://github.com/JOK9527/Pixel_Archive>  
> 生产分支：`main`  
> 部署平台：Cloudflare Workers Static Assets

## 1. 当前部署结构

Pixel Archive 是 Astro 纯静态站点，不依赖后端、数据库、KV 或 Session。

```txt
本地修改
  -> Git commit
  -> push 到 GitHub main
  -> Cloudflare 自动构建
  -> npm run build
  -> npx wrangler deploy
  -> 发布 dist/
```

关键配置：

| 项目 | 当前值 |
| --- | --- |
| Git 仓库 | `JOK9527/Pixel_Archive` |
| 生产分支 | `main` |
| Build command | `npm run build` |
| Deploy command | `npx wrangler deploy` |
| Root directory | `/` |
| 静态目录 | `dist` |
| 生产 URL 变量 | `PUBLIC_SITE_URL` |

`wrangler.jsonc` 只部署静态资源，不应添加 `SESSION`、KV 或 Worker 入口。

## 2. 日常更新标准流程

### 2.1 开始工作前

```powershell
cd E:\projects\Pixel_Archive
git status
git pull
npm install
```

`npm install` 在 `package.json` 或 `package-lock.json` 没变化时通常很快，也可以省略。

### 2.2 本地开发

```powershell
npm run dev
```

浏览器打开终端显示的本地地址，通常是：

```txt
http://localhost:4321
```

### 2.3 提交前检查

```powershell
npm run check
npm run build
git status
git diff
```

必须确认：

- `npm run check` 无错误。
- `npm run build` 成功。
- 没有提交 `.env`、`node_modules/`、`.astro/`、`dist/` 或本地输出。
- 内容中没有密钥、本机路径、邮箱、私人文档或未脱敏截图。

### 2.4 提交并发布

```powershell
git add <本次修改的文件>
git commit -m "类型: 简短说明"
git push origin main
```

常用提交类型：

```txt
feat: 新功能或新内容
fix: 修复问题
docs: 文档更新
style: 视觉和样式调整
refactor: 不改变功能的代码整理
chore: 构建或维护任务
```

推送后进入 Cloudflare：

```txt
Workers & Pages
  -> pixel-archive
  -> Deployments
```

确认最新构建对应刚推送的 commit，并显示成功。

## 3. 如何新增或更新内容

### 3.1 Projects

目录：

```txt
src/content/projects/
```

一个项目对应一个 `.md` 或 `.mdx` 文件。修改后重点检查：

- `status` 只能使用 `WIP`、`Done`、`Prototype`、`Paused`。
- `type` 必须符合 `src/content.config.ts`。
- `tags` 使用小写英文和连字符。
- `updatedDate` 写成 `YYYY-MM-DD`。
- 关联项必须使用目标内容的 slug。

### 3.2 Notes

目录：

```txt
src/content/notes/
```

建议文件名：

```txt
YYYY-MM-DD-short-slug.mdx
```

草稿阶段设置：

```yaml
draft: true
```

正式发布前改为：

```yaml
draft: false
```

### 3.3 Archive

目录：

```txt
src/content/archive/
```

适合记录里程碑、保存点和项目阶段日志。`type` 只能使用：

```txt
save-point
milestone
project-log
```

### 3.4 Lab

目录：

```txt
src/content/lab/
```

适合尚未成为正式功能的界面、视觉和交互实验。

### 3.5 图片

公开图片放在：

```txt
public/images/
```

内容中以站点根路径引用：

```md
![说明文字](/images/notes/example/image.svg)
```

不要引用本机绝对路径。截图上传前检查账号、窗口标题、地址栏、通知、聊天内容和文档正文。

## 4. Cloudflare 日常操作

### 4.1 修改环境变量

路径：

```txt
pixel-archive
  -> Settings
  -> Build
  -> Variables and secrets
```

当前必须保留：

```txt
PUBLIC_SITE_URL=https://pixel-archive.pixelarchive.workers.dev
```

它不是密钥，不需要 Encrypt。

修改环境变量不会自动重建。保存后需要重新部署，最简单的方式是推送一个正常提交；只有确实没有代码变化时才使用：

```powershell
git commit --allow-empty -m "chore: rebuild with updated environment"
git push origin main
```

### 4.2 GitHub 连接告警

如果出现“提取 GitHub 用户或组织详细信息时出错”：

1. 点击 Build 区域的 `Manage`。
2. 检查 Cloudflare Workers Builds GitHub App 是否仍有仓库权限。
3. 重新授权 `JOK9527/Pixel_Archive`。
4. 不要删除 Worker 或重新创建项目。

只要新 push 仍能触发构建，该告警可以暂时观察。

### 4.3 查看构建失败

进入：

```txt
Deployments -> 失败的 Build -> 日志底部
```

优先找第一条真实错误，不要只看最后的 `deploy command failed`。

常见情况：

| 现象 | 处理 |
| --- | --- |
| TypeScript 或 schema 错误 | 本地运行 `npm run check` |
| Astro build 失败 | 本地运行 `npm run build` |
| 再次创建 `SESSION` KV | 确认仓库中的 `wrangler.jsonc` 存在且没有 KV binding |
| GitHub 无法拉取 | 重新授权 GitHub App |
| 生产 URL 仍是旧地址 | 更新 `PUBLIC_SITE_URL` 后重新构建 |

### 4.4 回滚

推荐使用 Git 回滚，保证仓库和线上状态一致：

```powershell
git log --oneline
git revert <有问题的提交>
git push origin main
```

紧急情况下也可以在 Cloudflare `Versions` 中重新部署上一个成功版本，之后仍应在 Git 中补做回滚。

## 5. 修改域名或 workers.dev 子域

修改地址后必须同步更新：

```txt
PUBLIC_SITE_URL
```

然后重新构建。否则 canonical、Open Graph、RSS、sitemap 和 robots 仍会使用旧地址。

切换 `workers.dev` 子域时，DNS 和 SSL 证书可能需要几分钟生效，期间可能短暂出现：

```txt
ERR_SSL_VERSION_OR_CIPHER_MISMATCH
```

如果准备长期公开使用，建议后续绑定一个不含平台标识的自定义域名。

## 6. 发布后检查

每次重要更新后至少检查：

```txt
/
/projects/
/notes/
/archive/
/lab/
/rss.xml
/sitemap.xml
/robots.txt
/不存在的地址
```

同时检查：

- 桌面宽度。
- 390px 移动端。
- 浅色与暗色主题。
- 浏览器控制台无异常。
- favicon 已更新。
- 页面源代码中的 canonical 使用当前正式地址。

## 7. 性能说明

当前站点约 0.83 MB，共 70 个构建文件，首页 HTML 约 24 KB，没有外部字体、统计脚本或大型前端框架。

线上抽样结果：

```txt
DNS: 约 0.006 秒
TLS: 约 0.22-0.25 秒
首字节: 约 0.34-0.51 秒
首页请求总耗时: 约 0.45-0.72 秒
Cloudflare edge cache: HIT
边缘节点: NRT（东京）
```

因此当前主要延迟来自用户网络到 `workers.dev` 的链路，而不是页面代码或服务器计算。

项目中的 `public/_headers` 已设置：

- `/_astro/*`：一年 immutable 缓存。
- `favicon.svg`：七天缓存。
- `/images/*`：一天缓存并允许后台更新。
- HTML：继续使用 Cloudflare 默认重新验证，保证内容发布后及时更新。

如果在中国大陆网络中仍明显缓慢：

1. 分别使用手机流量和宽带测试，排除本地 DNS 或运营商链路。
2. 使用无痕窗口区分首次访问和缓存后的访问。
3. 后续绑定自定义域名，但自定义域名本身不保证改善大陆链路。
4. 若主要受众在中国大陆且要求稳定低延迟，需要单独评估大陆 CDN、备案和境内部署，不能仅靠前端压缩解决。

## 8. 不要做的操作

- 不要执行 `git push --force`。
- 不要执行 `npm audit fix --force`。
- 不要删除 `wrangler.jsonc`。
- 不要给纯静态站添加 `SESSION`、KV 或数据库。
- 不要直接编辑 `dist/`。
- 不要把密钥写入 `PUBLIC_` 环境变量。
- 不要在 Cloudflare 手动上传文件替代 Git 自动部署。

## 9. 最短操作清单

普通更新只需：

```powershell
git pull
npm run dev
# 修改内容
npm run check
npm run build
git add <文件>
git commit -m "feat: update content"
git push origin main
```

然后在 Cloudflare Deployments 中确认最新构建成功。
