# 01_规范总览指南

版本：v1.3  
项目：Pixel Archive  
位置：`docs/01_规范总览指南.md`  
用途：作为 Pixel Archive 全部规范文档与阶段计划文档的导航入口，说明每份文档的职责、阅读顺序、优先级、适用任务和冲突处理规则。  

---

## 1. 本文档的作用

本文档是 Pixel Archive 规范体系的总入口。

它负责回答：

```txt
1. 当前项目有哪些规范文档？
2. 每份文档负责什么？
3. Agent 或开发者在不同任务前应该读哪些文档？
4. 多份规范出现交叉时，以哪份为准？
5. 哪些文档是强约束，哪些只是参考？
6. 阶段性开发计划应该放在哪里？
7. 文档体系后续如何维护？
```

本文档不负责展开全部细则。具体细则应进入对应专项规范、阶段计划或参考文档查看。

## 3.5 第五层：阶段计划与阶段复盘

```txt

```

```txt

后续 Phase11 / Phase12 / Phase13 等阶段计划
阶段修正计划
阶段复盘文档
```

职责：

```txt
定义某一阶段的具体任务范围
记录当前阶段为什么要做
记录本阶段输入、输出、禁止事项和验收标准
把用户阶段性反馈转化为可执行计划
承接 16_阶段任务拆分与里程碑规范.md 之后的细化任务
```

注意：

```txt
phases/ 是阶段性执行文件，不是长期通用强规范。
phases/ 不得覆盖 00_项目开发计划书.md、01_规范总览指南.md 或 standards/ 中的强规范。
如果某一阶段计划沉淀出长期规则，应在阶段结束后提炼进 standards/ 中对应文档。
```

## 6. phases/ 的具体定位

## 6.1 `docs/phases/` 放什么

`docs/phases/` 用于存放阶段性开发计划、阶段修正计划和阶段复盘。

适合放入：

```txt
Phase10 视觉系统再校准计划
Phase11 真实内容填充计划
Phase12 搜索与阅读增强计划
Phase13 部署后优化计划
某阶段验收后的修正计划
某阶段完成后的复盘记录
```

不适合放入：

```txt
长期通用规范
技术栈原则
组件通用约束
数据模型契约
隐私边界
视觉总规范
参考资源清单
网站公开内容
```

---

## 6.2 phases/ 与 standards/ 的区别

```txt
standards/：长期有效，约束整个项目
phases/：阶段有效，约束某一轮开发
references/：只作参考，不主动触发实现
agent/：执行方式和汇报方式
src/content/：网站公开内容
```

判断规则：

```txt
会长期影响所有后续开发 → 放入 standards/
只服务某一阶段执行 → 放入 phases/
只是灵感、资料、资源池 → 放入 references/
给 Agent 复制执行 → 放入 agent/
会被网站公开展示 → 放入 src/content/
```

## 6.3 当前 Phase10 定位

当前新增阶段计划为：

```txt
docs/phases/18_Phase10_视觉系统再校准计划.md
```

它服务于 Phase 9 第一阶段整体验收之后的视觉修正，不改变第一阶段已经完成的结构基线。

主要处理：

```txt
紫色边框过量
像素风格表层化
暗色模式廉价感
首页两侧留白缺少空间秩序
色彩 token 缺少更克制的高级感
```

主要不处理：

```txt
搜索
RSS
评论
CMS
数据库
复杂像素房间
GitHub API 自动拉取
大规模真实内容填充
```

## 7. 规范优先级

当文档之间出现交叉或冲突时，按以下优先级处理：

```txt
1. 用户当前明确指令
2. 01_规范总览指南.md
3. 00_项目开发计划书.md
4. standards/ 中对应专项规范
5. phases/ 中当前阶段计划
6. agent/ 中执行规范和模板
7. references/ 中参考资料
8. Agent 自行判断
```

注意：

```txt
references/ 中的内容不能覆盖 standards/ 中的强规范。
phases/ 中的阶段计划不能覆盖 00–01 和 standards/ 中的长期约束。
若当前阶段计划与 16_阶段任务拆分与里程碑规范.md 冲突，应先询问用户确认阶段是否已经调整。
```

## 10. Agent 开发前检查

Agent 每次开发前应确认：

```txt
1. 当前任务属于哪个阶段？
2. 是否存在 docs/phases/ 中的当前阶段计划？
3. 是否超出当前阶段边界？
4. 需要读取哪些规范？
5. 是否涉及数据模型或内部接口？
6. 是否涉及公开信息或隐私风险？
7. 是否会新增依赖？
8. 是否会改变路由？
9. 是否会改变 Content Collections schema？
10. 是否会改变组件 props？
11. 是否会改变视觉主方向？
12. 是否需要先询问用户？
```

## 13. 规范维护规则

维护文档体系时应遵守：

```txt
1. 每条规则尽量只在主定义文档中详细定义
2. 其他文档只保留引用或摘要
3. 新增强规范时，应放入 standards/
4. 新增 Agent 相关模板时，应放入 agent/
5. 新增参考资料或未来想法时，应放入 references/
6. 新增阶段性开发计划、修正计划或复盘时，应放入 phases/
7. 修改目录结构后必须更新本文档
8. 修改阶段边界后必须更新 16_阶段任务拆分与里程碑规范.md 或新增 / 更新 docs/phases/ 对应阶段计划
9. 修改数据字段后必须更新 15_数据模型与内部接口规范.md
10. 涉及公开信息时必须检查 17_公开信息与隐私边界规范.md
11. 阶段计划中沉淀出的长期规则，应在阶段结束后提炼进 standards/ 中对应文档
```

## 14. 不应继续扩张的文档

当前 `00–17` 已经覆盖：

```txt
项目定位
技术执行
视觉风格
内容管理
模块架构
组件设计
文案风格
部署运维
验收测试
Agent 指令
Agent 工作流
视觉参考
技术参考
后续功能
数据模型
阶段里程碑
隐私边界
```

后续不建议继续随意新增长期强规范文件。

如果有新需求，优先判断能否补充到已有文档中。  
只有当某类规则长期独立、频繁使用、且会影响开发质量时，才考虑新增 standards 文档。

`docs/phases/` 是例外：阶段计划、阶段修正和阶段复盘可以随着开发推进继续新增，但它们不等同于新的长期强规范。

## 15. 最终原则

> Pixel Archive 的规范体系应当模块化、低重复、可执行。  
> 总览文档负责导航，专项规范负责规则，Agent 文档负责执行，references 只负责参考，phases 负责阶段推进。  
> 任何开发任务都应先找到对应规范和当前阶段计划，再开始实现。
