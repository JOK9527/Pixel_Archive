# 01_规范总览指南

版本：v1.2  
项目：Pixel Archive  
位置：`docs/01_规范总览指南.md`  
用途：作为 Pixel Archive 全部规范文档的导航入口，说明每份文档的职责、阅读顺序、优先级、适用任务和冲突处理规则。  

---

## 1. 本文档的作用

本文档是 Pixel Archive 规范体系的总入口。

它负责回答：

```txt
1. 当前项目有哪些规范文档？
2. 每份文档负责什么？
3. Agent 或开发者在不同任务前应该读哪些文档？
4. 多份规范出现交叉时，以哪份为准？
5. 哪些文档是强约束，哪些只是参考？
6. 文档体系后续如何维护？
```

本文档不负责展开全部细则。具体细则应进入对应专项规范文档查看。

## 5. 15–17 的具体定位

## 5.1 `15_数据模型与内部接口规范.md`

它是字段契约和内部接口规范。

主要约束：

```txt
ProjectItem
NoteItem
ArchiveItem
LabItem
status / type 枚举
frontmatter 与 schema 对齐
schema 与 src/lib/content 输出对齐
helper 输出与组件 props 对齐
slug / href 生成规则
navigation.ts
site.ts
CSS Variables 接口
内部接口变更流程
```

适用任务：

```txt
定义 Content Collections schema
新增 src/lib/content/*
实现 ProjectCard / NoteCard / ArchiveEntry / LabCard
修改 status / type 字段
修复字段名不一致问题
调整 slug、href、路由、导航、site 配置
```

---

## 5.2 `16_阶段任务拆分与里程碑规范.md`

它是阶段执行地图。

主要约束：

```txt
Phase 0：项目初始化
Phase 1：基础架构与视觉系统
Phase 2：首页初版
Phase 3：页面空壳与路由
Phase 4：Content Collections 与占位内容
Phase 5：Projects 列表与详情
Phase 6：Notes 列表与文章页
Phase 7：Archive / Lab / About / 404 Polish
Phase 8：部署准备
Phase 9：第一阶段整体验收与提交
```

适用任务：

```txt
判断现在应该做哪一步
判断任务是否超阶段
给 Agent 拆分开发任务
阶段验收
阶段 commit
```

---

## 5.3 `17_公开信息与隐私边界规范.md`

它是公开网站安全边界。

主要约束：

```txt
什么能公开
什么要谨慎公开
什么禁止公开
About 页面公开边界
Project 页面公开边界
Notes / Archive / Lab 公开边界
图片和截图检查
API Key / token / .env 处理
本地路径处理
他人信息处理
论文 / 学业资料公开边界
上线前隐私检查
```

适用任务：

```txt
准备公开网站
新增 About 内容
上传截图
迁移真实项目
公开学习笔记
公开论文 / 毕设 / 实验相关内容
部署上线前检查
```

## 6. 规范优先级

当文档之间出现交叉或冲突时，按以下优先级处理：

```txt
1. 用户当前明确指令
2. 01_规范总览指南.md
3. 00_项目开发计划书.md
4. standards/ 中对应专项规范
5. agent/ 中执行规范和模板
6. references/ 中参考资料
7. Agent 自行判断
```

注意：

```txt
references/ 中的内容不能覆盖 standards/ 中的强规范。
```

## 9. Agent 开发前检查

Agent 每次开发前应确认：

```txt
1. 当前任务属于哪个阶段？
2. 是否超出当前阶段边界？
3. 需要读取哪些规范？
4. 是否涉及数据模型或内部接口？
5. 是否涉及公开信息或隐私风险？
6. 是否会新增依赖？
7. 是否会改变路由？
8. 是否会改变 Content Collections schema？
9. 是否会改变组件 props？
10. 是否会改变视觉主方向？
11. 是否需要先询问用户？
```

## 11. 当前阶段边界

第一阶段重点：

```txt
Astro 项目初始化
TypeScript / MDX 配置
基础目录
CSS Variables 主题系统
BaseLayout
Header / Footer / ThemeToggle
首页 Archive Terminal 初版
页面空壳
Content Collections schema
规范占位内容
README / WORKLOG
npm run build 通过
```

第一阶段不做：

```txt
搜索
评论
CMS
数据库
后端
复杂像素头像
复杂像素房间
地图式首页
GitHub API 自动拉取
多语言
完整 SEO
Open Graph 图片系统
复杂动画
访问统计
联系表单后端
```

如需判断阶段边界，优先查看：


## 12. 规范维护规则

维护文档体系时应遵守：

```txt
1. 每条规则尽量只在主定义文档中详细定义
2. 其他文档只保留引用或摘要
3. 新增强规范时，应放入 standards/
4. 新增 Agent 相关模板时，应放入 agent/
5. 新增参考资料或未来想法时，应放入 references/
6. 修改目录结构后必须更新本文档
7. 修改阶段边界后必须更新 16_阶段任务拆分与里程碑规范.md
8. 修改数据字段后必须更新 15_数据模型与内部接口规范.md
9. 涉及公开信息时必须检查 17_公开信息与隐私边界规范.md
```

## 14. 最终原则

> Pixel Archive 的规范体系应当模块化、低重复、可执行。  
> 总览文档负责导航，专项规范负责规则，Agent 文档负责执行，references 只负责参考。  
> 任何开发任务都应先找到对应规范，再开始实现。
